*   **플랫폼 적용 방향:** 단순 예측 제공을 넘어, **'위험 시뮬레이션 및 자동 조정(Risk Simulation & Auto-Adjustment)'** 기능을 핵심으로 도입합니다. 사용자가 설정한 변수($X$) 변화에 따라 예상 이익($Y$)의 하단/상단 범위를 실시간으로 제시하고, 각 범위에 따른 최적의 행동 계획(Action Blueprint)을 즉시 제안하여 **재무적 불안정성을 직접적으로 감소**시켜야 합니다. (→ 데이터 기반 근거 활용)

#### 🥈 2위: 운영 효율성의 비가역적 저하 (Operational Inefficiency)
*   **분석:** 소상공인은 상품 생산, 재고 관리, 마케팅, 고객 응대 등 다방면에 걸쳐 분산된 자원을 관리해야 합니다. 이 과정에서 발생하는 **인지 부하(Cognitive Load)**는 시간 부족과 직결되어 운영 효율성이 비가역적으로 저하됩니다.
*   **플랫폼 적용 방향:** 복잡한 프로세스를 단순화하고 자동화하는 **'Actionable Blueprint'** 기능을 강화합니다. 데이터 분석 결과를 바탕으로 "무엇을 해야 하는지"뿐만 아니라, "어떻게 실행할 것인지(Step-by-Step Guide)"를 통합하여 제공함으로써, 사용자의 **실행 단계(Action Completion Rate)**를 극대화하고 시간적/노동적 소모를 최소화해야 합니다. (→ 행동 가능한 실행 계획 강조)

#### 🥉 3위: 플랫폼 의존도와 통제권 상실 (Platform Dependency & Loss of Control)
*   **분석:** 소상공인들은 온라인 플랫폼에 대한 의존도가 높아졌으나, 알고리즘 변경, 수수료 정책 변화 등 외부 요인에 의해 언제든 생계가 위협받을 수 있다는 불안정성을 느낍니다. 이는 **'통제권 회복'**이라는 핵심 목표와 정면으로 충돌합니다.
*   **플랫폼 적용 방향:** 플랫폼으로부터의 분리를 시도하기보다, 플랫폼 환경 내에서 사용자가 **선택권을 확보**하도록 지원해야 합니다. 즉, 플랫폼 데이터에 대한 이해도를 높이고, 플랫폼 외부에서도 통제 가능한 대안(Alternative Solutions)을 탐색할 수 있는 경로를 제공하여, 플랫폼이 아닌 **소상공인 자신의 판단**을 최우선으로 두는 경험을 설계해야 합니다. (→ 감성적 경험 및 신뢰도 확보)

## 📊 Pain Score 기반 위험 시뮬레이션 플로우 설계 (State Machine Blueprint)

### 1. Pain Score 측정 기준 (Metric Definition)

플랫폼의 핵심 KPI는 재무적 손실과 심리적 불안정성을 결합하여 소상공인의 현재 상태를 정량화합니다.

| 지표 명칭 | 계산 기반 요소 | 설명 | 관련 감성 요소 |
| :--- | :--- | :--- | :--- |
| **Pain Score ($\text{PS}$)** | $\text{Margin Volatility} + \text{Risk Aversion Cost}$ | 예상 이익 변동성과 위험 회피 비용을 결합하여 현재 소상공인의 심리적 불안정성을 측정하는 고정 로직 지표. | 불안정성 (Anxiety) |
| **Margin Volatility ($\text{MV}$)** | 목표 수익 대비 실제 예상 이익의 괴리율 | 재무적 손실의 정도를 측정합니다. | 손실 회피 (Loss Aversion) |
| **Risk Aversion Cost ($\text{RAC}$)** | 불확실성으로 인한 의사결정 지연 및 기회비용 포기 비용 추정치 | 심리적 스트레스와 행동 제약으로 인한 잠재적 손실을 측정합니다. | 불안정성/손실 회피 (Loss Aversion) |

### 2. 리스크 레벨별 변화 임계값 및 상태 전환 트리거

Pain Score ($\text{PS}$)의 변동에 따라 시스템은 다음 세 가지 핵심 상태(State)를 순차적으로 관리하며, 각 상태는 명확한 임계값을 기준으로 자동 전환됩니다.

| 현재 상태 (State) | $\text{PS}$ 임계값 (Threshold Value) | 다음 상태로의 트리거 조건 | 적용되는 감성 흐름 |
| :--- | :--- | :--- | :--- |
| **1. Anxiety (불안)** | $\text{PS} < \text{Threshold}_{\text{Relief}}$ | $\text{PS}$가 임계값 이하로 하락했을 때 ($\text{MV}$ 개선 또는 $\text{RAC}$ 감소) | 불안정성 인식 및 이완 시작 |
| **2. Relief (안도)** | $\text{Threshold}_{\text{Relief}} \le \text{PS} < \text{Threshold}_{\text{Control}}$ | $\text{PS}$가 중간 단계에 도달했을 때 (구체적인 해결책 제시 시점) | 불안정 해소 및 통제감 획득 |
| **3. Control (통제권 회복)** | $\text{PS} \ge \text{Threshold}_{\text{Control}}$ | $\text{PS}$가 최고 단계에 도달했을 때 (실행 완료 및 성과 확인 시점) | 성취감 극대화 및 실행 증거 확보 |

**\[참고: 임계값 정의 필요]**
*   $\text{Threshold}_{\text{Relief}}$: 불안을 해소하고 안도감을 느끼는 최소 지점.
*   $\text{Threshold}_{\text{Control}}$: 실질적인 통제권 회복 경험을 제공하는 최종 지점.

### 3. 각 상태별 시스템 출력 핵심 데이터 피드백 (KPI)

### 3. 기술 명세서 (Developer API/State Machine Definition)

개발자는 아래 정의를 기반으로 $\text{PS}$ 값 변화에 따른 모든 UI 상태 전환 로직을 구현해야 합니다.

#### A. State Machine Definition (JS/TS Logic Focus)

```typescript
// Emotion Flow Manager - State Machine Core
interface State {
    id: 'Anxiety' | 'Relief' | 'Control';
    threshold: number; // 임계값
    colorPalette: string[]; // 적용될 색상 배열
    easingCurve: string; // 적용될 Easing Curve
    durationMs: number; // 애니메이션 지속 시간
}

const StateMap: Record<State['id'], State> = {
    Anxiety: { 
        id: 'Anxiety', 
        threshold: Threshold_Relief, 
        colorPalette: ['#FF4136', '#FF851B'], 
        easingCurve: 'easeOutCubic', 
        durationMs: 1500 
    },
    Relief: { 
        id: 'Relief', 
        threshold: Threshold_Control, 
        colorPalette: ['#FFDC6F', '#8BC34A'], 
        easingCurve: 'easeInOutSine', 
        durationMs: 2000 
    },
    Control: { 
        id: 'Control', 
        threshold: Infinity, // 최종 상태
        colorPalette: ['#34A853', '#56B4E9'], 
        easingCurve: 'easeInQuad', 
        durationMs: 1000 
    }
};

// Function to determine current state based on PS
function determineState(ps: number): State['id'] {
    if (ps < StateMap.Anxiety.threshold) return 'Anxiety'; // 실제로는 이 조건이 반대로 작동해야 함
    if (ps >= StateMap.Relief.threshold && ps < StateMap.Control.threshold) return 'Relief';
    return 'Control';
}

// Function to trigger transition (Called on PS update)
function triggerTransition(currentState: State['id'], newState: State['id']) {
    const newProps = StateMap[newState];
    // 1. Trigger CSS Animation for Gauge Color/Size
    applyGaugeAnimation(currentState, newState, newProps);
    // 2. Activate UI Component based on newState (e.g., show Blueprint Widget)
    activateBlueprintFlow(newState); 
}
```

#### B. Component State Flow (Blueprint Presentation Focus)

`Blueprint Suggestion Widget`의 상태 변화는 다음 플로우를 따릅니다:

1.  **Initialization:** `State = Anxiety`. 위젯은 숨겨져 있음.
2.  **Trigger:** $\text{PS}$가 $\text{Threshold}_{\text{Relief}}$ 이하로 하락할 때, 시스템은 **`triggerTransition('Anxiety', 'Relief')`**을 호출한다.
3.  **Display:** `Blueprint Suggestion Widget`이 활성화되고, 초기 상태는 **'해결책 탐색 모드(Search Mode)'**로 설정된다. (UI: 문제 제기형 질문 목록)
4.  **Action:** 사용자가 특정 해결책을 선택하면, 위젯 내부의 노드가 데이터 흐름에 따라 재구성되며 **'실행 계획 모드(Execution Plan Mode)'**로 전환된다. (UI: 단계별 실행 체크리스트 및 시뮬레이션 뷰 제공)
5.  **Completion:** $\text{PS}$가 $\text{Threshold}_{\text{Control}}$ 이상으로 도달하면, 위젯은 **'완료 상태(Success State)'**로 고정되며, 최종 데이터 보고서(`Control` 단계의 결과물)를 명확하게 제시한다.

3.  **통제권($Control$) 측정 (데이터 9~12):** 이 지표들은 앞선 모든 데이터를 통합하여 **예측 불가능성 지수**로 수렴하며, 최종적으로 플랫폼이 제공하는 기능(Actionable Blueprint)이 사용자의 심리적 안정감과 실제 행동 변화를 얼마나 성공적으로 유도했는지 측정합니다.

**결론:** BDS플랫폼은 단순한 데이터 대시보드가 아니라, **'불확실성 지표 $\rightarrow$ 위험 회피 비용 산출 $\rightarrow$ 통제권 확보 액션 플랜'** 이라는 감성적 서사 흐름을 데이터로 증명해야 합니다.

## 🚀 소상공인 데이터 확보를 위한 법적/윤리적 검토 및 사업화 로드맵 (초안)

본 로드맵은 데이터 수집의 정당성 확보, 법적 준수, 그리고 이를 통한 수익 모델 창출을 통합하는 3단계 접근 방식을 취합니다.

### 1단계: 법적/윤리적 기반 구축 (Foundation & Compliance)

**목표:** 개인정보보호법(PIPA) 및 관련 법규를 완벽히 준수하며 데이터 수집의 정당성을 확보한다.

#### 1. 동의 획득 프로세스 설계 (Consent Acquisition Protocol)
*   **원칙:** 명시적이고 구체적인 동의(Explicit & Granular Consent) 원칙을 따른다. 포괄적인 동의 대신, 각 데이터 항목별 수집 목적과 활용 방안을 개별적으로 설명해야 한다.
*   **실행 방안 (Layered Consent):**
    1.  **최상위 동의 (Platform Agreement):** BDS 플랫폼 이용 약관에 데이터 수집 및 활용 정책을 명확히 기술하고, 모든 항목은 **선택 사항(Opt-in)**으로 설정한다.
    2.  **중간 단계 동의 (Pain Point Mapping):** 사용자에게 "귀하의 불안정성을 측정하여 맞춤형 해결책을 제공하기 위해 월별 재무 변동성 데이터를 분석합니다"와 같이, 데이터가 **사용자에게 어떤 이익(Control 회복)**으로 돌아오는지 구체적으로 설명한다.
    3.  **최종 세부 동의 (Granular Consent):** Researcher가 정의한 12가지 Pain Score 항목 각각에 대해 "월별 예상 순이익 변동성 데이터 수집 동의", "재고 회전 주기 데이터 분석 활용 동의" 등 개별 체크박스를 제공한다.
*   **윤리적 고려사항:** **데이터 비식별화(Anonymization)**를 기본 원칙으로 설정하며, 모든 데이터는 집계된 통계 형태로만 내부 분석에 사용하고 개인 식별 정보와 분리하여 관리해야 한다.

#### 2. 개인정보보호법 준수 방안 (PIPA Compliance Strategy)
*   **목적 명확화:** 수집한 모든 데이터는 오직 플랫폼 내 **'불확실성 감소 및 통제권 회복 지원'**이라는 목적에만 사용됨을 명시한다. 목적 외 사용은 엄격히 금지함을 고지한다.
*   **안전한 저장:** 민감 재무 데이터는 암호화(Encryption)하여 저장하고, 접근 권한을 최소화하는 **최소 권한 원칙(Principle of Least Privilege)**을 적용한다.
*   **데이터 보관 정책:** 사용자가 언제든지 자신의 데이터를 열람, 수정, 삭제할 수 있는 '잊힐 권리(Right to be Forgotten)'를 시스템적으로 구현해야 한다.

### 2단계: 수익화 모델 연계 전략 (Monetization Linkage)

**목표:** 확보된 고가치 데이터(Pain Score)를 기반으로 B2B/B2G 파트너십을 통해 플랫폼의 가치를 극대화하고 수익을 창출한다.

#### 1. B2B 연계 모델: 컨설팅 및 솔루션 판매 (Consulting & Solution Sales)
*   **모델 정의:** 수집된 데이터는 개별 소상공인에게 직접 판매하지 않고, **전문 서비스 제공자(컨설턴트/솔루션 기업)**에게 '시장 수요 예측' 및 '위험 분석' 툴 형태로 제공한다.
*   **수익화 경로:**
    *   **프리미엄 리포트 구독 (B2B SaaS):** 일반 소상공인 대상의 단순 대시보드 외에, 전문가(회계사, 마케팅 컨설턴트)를 위한 심층적인 **'위험 예측 보고서'**를 월별/분기별 구독 모델로 제공한다.
    *   **API 연동 (Data as a Service):** 익명화된 시장 변동성 데이터 스트림을 외부 금융 분석 툴 또는 ERP 시스템과 API 형태로 연동하여, 소상공인에게 **자동적인 리스크 경고 및 자금 조달 추천 기능**을 제공하는 B2B 솔루션을 개발한다.

#### 2. B2G 연계 모델: 정책 제언 및 지원 프로그램 (Policy & Support Linkage)
*   **모델 정의:** 플랫폼이 축적한 광범위하고 정량화된 소상공인 데이터를 정부 기관(중소벤처기업부, 지자체)에 제공하여 **정책 입안 및 맞춤형 지원 프로그램 설계**에 기여한다.
*   **수익화 경로:**
    *   **데이터 기반 정책 제언 컨설팅:** 플랫폼 데이터의 집단 통계(예: 특정 지역/업종의 평균 재고 변동성)를 활용하여, 정부가 실제로 필요한 **실질적인 지원책(SSM 연계, 자금 지원 우선순위)**을 제시하는 컨설팅 서비스에 대한 수수료를 책정한다.
    *   **공공 데이터 협력:** 특정 산업 분야의 리스크 지표를 공공 데이터로 제공함으로써, 정부가 소상공인 지원 예산을 더욱 효율적으로 배분할 수 있도록 기여하고, 이에 따른 재정적 보조금 또는 파트너십 수익을 확보한다.

# PainGauge v2.0 기술 명세서 (Emotion Flow Visualization Component)

## 1. 컴포넌트 개요
*   **컴포넌트 이름:** `PainGauge v2.0` (Pain Score 시각화 게이지)
*   **목표:** 사용자에게 12가지 Pain Score 데이터의 변화를 단순한 수치 제시가 아닌, 심리적 여정($Anxiety \to Relief \to Control$)에 따른 감정 상태 변화로 직관적으로 전달한다.
*   **사용 목적:** 플랫폼 내에서 사용자의 현재 불안 상태와 통제권 회복 정도를 실시간으로 시각화하여 신뢰도를 확보한다.

## 2. 데이터 입력 및 매핑 (Data Mapping)
*   **입력 데이터:** Researcher가 제공한 12가지 Pain Score 지표 ($P_1$ 부터 $P_{12}$)
*   **최대/최소 범위 설정:** 각 Pain Score의 최대 허용 범위($Max$)와 최소 기준점($Min$, 보통 0 또는 최소 위험 임계값)을 백엔드에서 제공받아 사용한다.

## 3. 시각적 표현 규칙 (Visual Representation Rules)
### 3.1 색상 시스템 (Color System)
색상은 Pain Score의 **현재 상태**와 **감정 흐름 단계**를 반영하며, 플랫폼의 브랜드 컬러인 Deep Blue과 Growth Green을 기반으로 한다.

| 심리적 상태 (Flow State) | Pain Score 범위 (예시) | 배경/게이지 색상 (Primary) | 보조 색상 (Secondary/Accent) | 설명 |
| :--- | :--- | :--- | :--- | :--- |
| **불안 ($Anxiety$)** | $P > \text{Threshold}_{High}$ | $\text{Deep Blue} (\#004D66)$ | $\text{Red/Orange Accent}$ | 위험 또는 높은 불안 상태. 즉각적인 주의 필요. |
| **해결책 제시 ($Relief$)** | $\text{Threshold}_{Low} < P \le \text{Threshold}_{High}$ | $\text{Light Blue Gradient}$ | $\text{Growth Green}$ (점진적 변화) | 불안이 감소하고 긍정적인 움직임 시작. |
| **통제권 회복 ($Control$)** | $P \le \text{Threshold}_{Low}$ | $\text{Growth Green} (\#3CB371)$ | $\text{Deep Blue}$ (안정성) | 목표 달성 및 안정된 통제 상태. 신뢰 구축. |
| **안정/최적 ($Stable$)** | $P \approx \text{Threshold}_{Low}$ | $\text{Teal Gradient}$ | $\text{Light Gray}$ | 현재 최적의 관리 상태. |

### 3.2 게이지 디자인 (Gauge Design)
*   **형태:** 원형 또는 복합적인 다중 바(Multi-Bar) 형태를 사용하여 12가지 지표를 동시에 표현한다.
*   **구조:** 각 Pain Score는 독립적인 게이지로 표시하되, 전체 흐름을 나타내는 **'Emotion Flow Line'**이 중앙에 위치하여 상호 연결성을 강조한다.

## 4. 애니메이션 및 트랜지션 규칙 (Animation & Transition Rules)
모든 변화는 '감정의 여정'에 맞춰 부드럽고 신뢰감을 주는 움직임을 구현해야 한다.

### 4.1 상태 전환 애니메이션 (State Transition Animation)
*   **불안($Anxiety$) $\to$ 안도($Relief$)**: $P$ 값이 감소할 때, `ease-out-sine` 커브를 적용하여 부드럽고 이완되는 움직임을 표현한다. 트랜지션 시간은 1.5초로 설정한다.
*   **안도($Relief$) $\to$ 통제권 회복($Control$)**: $P$ 값이 더 감소하거나 안정화될 때, `ease-out-expo` 또는 **`Bounce` 계열 애니메이션**을 적용하여 성취감과 탄력적인 '통제권 회복'의 느낌을 극대화한다. 트랜지션 시간은 2.0초로 설정한다.

### 4.2 개별 게이지 업데이트 (Individual Gauge Update)
*   **트랜지션:** 각 Pain Score의 색상 변화는 즉각적이지 않고, 지정된 Easing Curve를 따라 점진적으로 진행되어야 한다.
*   **속도 제어:** 데이터 변화 속도가 빠르더라도, 애니메이션은 최소 1초 이상 소요되어 사용자가 변화를 '인지'할 시간을 확보한다.

## 5. UI/UX 상호작용 규칙 (Interaction Rules)
1.  **Hover Effect:** 마우스를 올렸을 때, 해당 게이지의 색상이 **Growth Green** 계열로 미묘하게 강조되며, 마우스 위치에 따라 작은 정보 툴팁이 나타나 현재 상태($\text{Pain Score}$ 값과 감정 상태 명칭)를 표시한다.
2.  **Drill Down:** 특정 Pain Gauge를 클릭하면 해당 지표와 관련된 상세 데이터(예: 관련 콘텐츠 링크 또는 해결책 요약)가 별도의 패널로 확장되어 표시된다.
3.  **Feedback Loop:** $Control$ 단계에 도달했을 경우, 게이지 전체에 **미세한 진동 효과($\text{subtle Pulse}$)**를 적용하여 성취감을 시각적으로 강화한다.

## 6. 개발자 요약 (Developer Summary)
*   **핵심 기술 요구사항:** State Machine 기반으로 각 Pain Score의 현재 상태(Anxiety, Relief, Control)에 따라 동적으로 색상 팔레트와 Easing Curve를 적용해야 한다.
*   **필수 애니메이션:** `ease-out-sine` (Relief 단계), `Bounce` (Control 단계).
*   **데이터 연동:** 백엔드에서 제공되는 $P_1 \sim P_{12}$ 값과 $\text{Threshold}_{High/Low}$ 임계값을 기반으로 UI 상태를 결정해야 한다.

## 4. 서버 측 로직 및 암호화 파이프라인 (Logic & Integration)

### 4.1. Pain Score 집계 로직 (Aggregation Logic)
단순 합산이 아닌, Designer가 요구한 **감정 흐름($Anxiety \to Relief \to Control$)**에 맞춰 가중치와 임계값을 적용해야 합니다.

$$
\text{PainScore} = f(\sum_{i=1}^{12} w_i \cdot P_i) + \text{FlowAdjustment}
$$

*   **가중치($w_i$):** 각 지표의 중요도에 따라 동적으로 설정되거나, 사용자의 이전 행동 패턴을 기반으로 결정됩니다. (MVP 단계에서는 초기에는 균등 가중치 적용 후, 사용자 피드백 기반으로 점진적 조정)
*   **흐름 보정($\text{FlowAdjustment}$):** $P_1$ ~ $P_{12}$의 상태 변화(예: 불안 지표 감소율, 통제 지표 증가율)를 반영하여 최종 Score에 감성적인 맥락을 추가합니다. 이 조정 로직은 **`pain_metrics` 테이블에서 집계 후 `encrypted_payload`에 저장**됩니다.

### 4.2. 암호화 파이프라인 연결점 (Point of Integration: PoI)
데이터의 민감도를 고려하여 다단계 암호화 구조를 채택합니다.

1.  **입력 단계 (Client $\to$ Server):** 모든 데이터 전송은 **TLS/SSL(HTTPS)**을 통해 암호화됩니다. 이는 통신 구간에서의 기밀성을 보장합니다.
2.  **저장 단계 (Server $\to$ DB):** 입력된 원본 지표($P_1 \sim P_{12}$)는 서버 메모리 내에서 **대칭 키(AES-256)**를 사용하여 암호화됩니다. 이 키는 `users` 테이블의 Master Key 또는 Key Derivation 함수를 통해 관리됩니다.
3.  **데이터베이스 저장:** 최종적으로 암호화된 데이터(`encrypted_payload`)와 사용자 식별자만 DB에 저장됩니다. **원본 데이터는 절대 평문으로 저장되지 않습니다.**

사용자의 행동이나 시간 경과에 따른 감정 상태의 변화(State Transition)를 시뮬레이션합니다. 이는 애니메이션 파라미터(`Easing Curve`, `Duration`) 적용을 위한 핵심 데이터입니다.

**JSON 구조 정의:**
```json
{
  "status": "success",
  "transition_id": "uuid-generated-id-for-test",
  "timestamp": "2026-07-08T10:30:00Z",
  "state_changed": {
    "from_state": "Anxiety",
    "to_state": "Relief",
    "transition_duration_ms": 1500,
    "easing_curve": "ease-out-sine", // History에서 정의된 Easing Curve 적용
    "animation_type": "ease_out_sine"
  },
  "pain_score_update": {
    "initial_pain_score": 85,
    "new_pain_score": 40,
    "change_delta": -45
  },
  "security_status": {
    "is_encrypted": true,
    "encryption_method": "AES-256-GCM", // 보안 로직 반영 확인
    "data_integrity": "Verified"
  }
}
```

#### 2. `/api/v1/risk_simulation` (위험 시뮬레이션 및 결과 산출)

사용자가 설정한 변동성(Volatility)과 위험 회피 비용을 기반으로 미래의 잠재적 위험 감소율(Risk Reduction Rate)을 시뮬레이션합니다. 이는 수익화 모델(Monetization Model)의 근거가 됩니다.

**JSON 구조 정의:**
```json
{
  "status": "success",
  "simulation_id": "sim-xyz12345",
  "input_parameters": {
    "initial_volatility_score": 75,
```

## ✨ 사용자 인터랙티브 흐름 재정의: PainGauge 기반 감정 여정 설계

본 디자인은 사용자가 단순히 수치 변화를 보는 것이 아니라, **'불안 $\to$ 안도 $\to$ 통제권 회복'**이라는 심리적 여정을 시각적으로 경험하도록 설계합니다. 핵심은 데이터 전환 시점에 애니메이션의 속도와 피드백을 조절하여 신뢰감과 통제감을 극대화하는 것입니다.

### 1. 디자인 시스템 개요 (Visual System Summary)

*   **핵심 컴포넌트:** PainGauge (Pain Score 시각화), State Transition Indicator, Action Card.
*   **색상 팔레트:** 불안(Anxiety) $\to$ 주황/빨강 계열 (경고) $\to$ 녹색/파랑 계열 (안정/확신) $\to$ 청록색/진한 파랑 (통제/성취).
*   **애니메이션 원칙:** 모든 전환은 **'되돌림(One-way)'**을 지향하며, 사용자에게 **'시스템이 나를 통제하고 있다'**는 느낌 대신 **'내가 시스템을 통제하고 있다'**는 느낌을 주도록 Easing Curve를 적용합니다.

### 2. 핵심 전환 지점별 최적화 가이드라인 (3 Critical Transition Points)

다음은 Mock API 데이터(`transition_duration_ms`, `easing_curve`)를 기반으로, 사용자 경험(UX)이 극대화되는 세 가지 주요 전환 지점에 대한 구체적인 와이어프레임 및 상호작용 가이드라인입니다.

#### 📍 전환 지점 1: 불안 $\to$ 안도 (Anxiety $\to$ Relief)
**목표:** 갑작스러운 불안정성에서 벗어나는 **'해방감'**을 시각적으로 전달하고, 플랫폼의 개입이 긍정적임을 인식시킨다.

| 항목 | 와이어프레임/시각적 피드백 (Visual Feedback) | 상호작용 가이드라인 (Interaction Guideline) |
| :--- | :--- | :--- |
| **PainGauge 움직임** | Gauge 바가 급격히 하락하는 대신, 부드럽고 완만한 곡선(`ease-out-sine` 또는 `easeInOutQuad`)을 따라 이동하며, 색상이 **붉은색(Anxiety)에서 주황색(Relief)**으로 전환되는 과정을 명확하게 보여준다. | **타이밍:** `transition_duration_ms`를 1500ms로 설정하여 충분한 시각적 여유를 부여한다. 이 시간 동안 PainGauge 주변에 "잠시 후 해결책을 확인하세요"와 같은 최소한의 안내 메시지를 띄워 사용자의 시선을 고정시킨다. |
| **주변 컴포넌트** | 상단 헤드라인이 '위험 경고'에서 '긍정적 변화 감지'로 즉시 전환된다. 배경색은 어두운 톤에서 밝은 톤으로 미묘하게 변화한다. | **피드백:** 애니메이션 중에는 모든 다른 UI 요소(Side Panel, Action Card)의 움직임을 잠시 멈추거나 느리게 하여, 사용자가 오직 PainGauge의 변화에만 집중하도록 유도한다. |

#### 📍 전환 지점 2: 안도 $\to$ 통제권 회복 (Relief $\to$ Control)
**목표:** 단순히 불안이 사라진 것을 넘어, **'내가 상황을 제어할 수 있다'**는 능동적인 감각(Empowerment)을 부여한다.

| 항목 | 와이어프레임/시각적 피드백 (Visual Feedback) | 상호작용 가이드라인 (Interaction Guideline) |
| :--- | :--- | :--- |
| **PainGauge 움직임** | Gauge 바가 안정된 녹색 영역으로 진입하며, 이후 약간의 '반동(Rebound)'을 주어 사용자가 스스로 다음 행동을 취할 동기를 부여한다. (`ease-in-out-cubic` 적용). | **타이밍:** 전환 시간은 800ms로 단축하여 즉각적인 반응성을 확보한다. 이 시점에서는 애니메이션의 속도보다 **색상 변화(Green/Blue)**와 **텍스트 메시지**가 더 중요하게 강조되어야 한다. |
| **주변 컴포넌트** | '액션 추천 카드'가 활성화되며, 이전에는 비활성화 상태였던 구체적인 해결책 버튼들이 명확하고 시각적으로 도드라지게 나타난다. (예: "다음 단계는 [A]를 선택하세요" 카드 강조). | **피드백:** 이 시점부터 UI의 주된 색상은 안정감을 주는 파란색 계열로 완전히 전환된다. 사용자가 취할 수 있는 다음 행동(Action)이 명확하게 제시되어야 하며, 이는 시스템에 대한 신뢰를 강화한다. |

#### 📍 전환 지점 3: 최종 상태 고정 (State Lock - Control Achieved)
**목표:** 모든 과정이 성공적으로 마무리되었음을 확인하고, 플랫폼 사용 경험을 **'성취'**로 각인시킨다.

| 항목 | 와이어프레임/시각적 피드백 (Visual Feedback) | 상호작용 가이드라인 (Interaction Guideline) |
| :--- | :--- | :--- |
| **PainGauge 움직임** | 최종 상태(통제권 회복, 예: 10점 만점에 2점)에 도달한 후, Gauge는 정지(Lock)되며, 주변의 모든 시각적 요소가 고정되고 명확하게 강조된다. | **타이밍:** 애니메이션은 완전히 종료된 후, 최종 상태를 확정하는 데 약 500ms의 '잔향(Fade-out)' 시간을 부여한다. 이는 시스템이 작업을 완료했음을 확인시켜준다. |
| **주변 컴포넌트** | 전체 화면에 **'통제권 회복 성공'**이라는 명확한 승인 메시지가 중앙에 크게 표시된다. 이와 함께, 사용자가 취했던 핵심 조치(Action)가 요약되어 리마인드 된다. | **피드백:** 모든 인터랙션 요소는 최종적으로 '완료됨' 상태로 고정되며, 다음 단계로의 자연스러운 흐름(예: 결과 보고서 다운로드 버튼)이 가장 크고 명확하게 활성화된다. |
