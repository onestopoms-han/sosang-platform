## 📑 [QA/Dev] Loss-Action Loop & 수익 모델 통합 테스트 케이스 우선순위 (V1.0)
**목표:** 핵심 매출 흐름(Paywall)과 사용자 경험(Loop 완성도)의 결함이 발생하지 않도록 검증한다.
**기준:** 비즈니스 가치 손실 위험도가 높은 순서대로 배정 (P0 > P1 > P2).

### 🚀 **P0: Critical Path (Critical - 매출 및 핵심 로직)**
*   **FAIL 시 영향도:** 치명적. 제품의 핵심 기능 또는 수익 창출이 불가능함. 즉시 수정 필요.
*   **테스트 목표:** 사용자가 Loss를 인지하고, 이를 해결하기 위해 Action Plan을 구매하며(Paywall), 실제로 개선된 상태(Gain)에 도달하는 전체 흐름의 무결성 검증.

| 테스트 케이스 ID | 기능 영역 | 구체적 시나리오 (Test Case) | 비즈니스 리스크 커버리지 |
| :--- | :--- | :--- | :--- |
| **P0-1** | **Paywall & 결제 게이트** | 1. 유료 콘텐츠 접근 시, 정확한 Loss Gauge Red/Green 상태와 함께 Paywall이 노출되어야 한다. <br> 2. 구독 만료 또는 트라이얼 기간 종료 시, 즉시 유료 전환(Action Required) 메시지가 나타나야 한다. (재구매 흐름 검증). | **핵심 수익 로직 누수 방지.** Pricing_Strategy_Final.md의 가격 정책이 정확히 적용되는가? |
| **P0-2** | **상태 전이(State Transition)** | 1. 사용자가 유료 액션을 수행하면, 시스템은 Loss 상태 $\rightarrow$ Action Plan 생성 $\rightarrow$ Gain 상태로 성공적으로 전환되어야 한다. <br> 2. 무료 사용자에게는 '다음 행동'을 위한 강력한 유도 문구(Loss Gauge)가 지속적으로 표시되어야 한다. | **Loss-Action Loop의 완성도 검증.** 사용자가 서비스에 머무르며 다음 단계 구매를 결정하게 만드는 핵심 흐름이 작동하는가? |
| **P0-3** | **API 결함 대응** | 1. 백엔드 API 호출 실패(예: 네트워크 오류) 시, 사용자에게 "오류 발생" 대신 "잠시 후 다시 시도해주세요"와 같은 친절한 안내 메시지를 제공하고 재시도를 유도해야 한다. (UX 안정성). | **시스템 장애로 인한 고객 이탈 방지.** 기술적 결함이 브랜드 신뢰도로 연결되지 않게 관리하는가? |

### ✨ **P1: High Priority (Must-Have - 주요 사용자 경험 및 데이터)**
*   **FAIL 시 영향도:** 높음. 사용자가 불편함을 느끼거나, 가치를 체감하지 못하여 이탈할 가능성이 큼. 즉시 개선 필요.
*   **테스트 목표:** 초기 사용자 온보딩 경험과 핵심 지표(KPI) 추적의 정확성 검증.

| 테스트 케이스 ID | 기능 영역 | 구체적 시나리오 (Test Case) | 비즈니스 리스크 커버리지 |
| :--- | :--- | :--- | :--- |
| **P1-1** | **Loss Gauge 측정 로직** | 1. 소상공인이 입력한 데이터(예: 매출 하락률, 특정 비용 증가율 등)가 Loss Gauge에 반영될 때, 시각적 임계치(Threshold)를 정확히 초과하여 빨간색으로 표시되어야 한다. <br> 2. 'Action Plan'을 실행했을 때, 시간이 지남에 따라 Loss Gauge의 색상이 점진적으로 개선되는 애니메이션/표시가 작동해야 한다. | **제품 가치 체감도 증명.** 플랫폼이 사용자에게 실질적인 위협(Loss)과 해소(Gain)를 시각적으로 잘 보여주는가? |
| **P1-2** | **사용자 데이터 저장성** | 1. 사용자가 세션을 종료했다가 재접속했을 때, 이전의 Loss Gauge 값, 생성된 Action Plan 초안 등이 유지되어야 한다. (로그인 기반 상태 관리). | **서비스 연속성과 신뢰도 확보.** 고객이 자신의 데이터를 안전하게 믿고 서비스에 의존할 수 있도록 하는가? |
| **P1-3** | **액션 플랜 실행 검증** | 1. 'A 액션'을 수행했다고 체크한 후, 다음 단계에서 해당 액션의 결과(예: "광고비 재분배로 효율 개선")를 구체적인 수치나 가이드를 통해 보여줘야 한다. (단순 클릭 방지). | **구체적 행동 유도.** 사용자가 '무엇을 해야 할지' 막연하게 느끼는 것을 넘어, '어떻게 해야 하는지'에 대한 답을 주는가? |

### 🟡 **P2: Medium Priority (Nice-to-Have - 예외 처리 및 확장성)**
*   **FAIL 시 영향도:** 낮음. 사용 경험의 완성도를 높이거나, 사소한 불편함을 유발할 수 있음. 다음 스프린트에서 개선 가능.
*   **테스트 목표:** 다양한 사용자 경로와 시스템 예외 상황 대응.

| 테스트 케이스 ID | 기능 영역 | 구체적 시나리오 (Test Case) | 비즈니스 리스크 커버리지 |
| :--- | :--- | :--- | :--- |
| **P2-1** | **데이터 입력 유효성 검증** | 1. Loss Gauge에 필요한 데이터 필드(예: 월 매출액, 인건비 등)를 누락하거나 비정상적인 값(음수 등)을 입력했을 때, 시스템이 경고 메시지를 통해 적절히 막아줘야 한다. | **데이터 무결성 확보.** 잘못된 데이터가 쌓여 AI 분석의 근거 자료로 오염되는 것을 방지하는가? |
| **P2-2** | **다국어/접근성 테스트** | 1. (향후 확장 시) 서비스 언어를 변경했을 때 모든 UI 요소와 메시지가 정확하게 번역되어 노출되는지 확인한다. <br> 2. 스크린 리더를 사용하는 사용자를 위해 모든 상호작용 요소에 적절한 ARIA 레이블이 적용되었는지 검사한다. | **시장 확장성 및 포용적 디자인.** 글로벌 시장 또는 약자 사용자층을 고려하는 준비 단계. |

---
**🔑 핵심 요약 (개발팀에게 전달할 메시지):**
테스트의 최우선 순위는 **P0-1 (Paywall)**과 **P0-2 (Loop 완성도)**입니다. 이 두 흐름이 단 한 번의 오류로 무너지면, 우리의 비즈니스 모델 전체가 붕괴됩니다. 개발팀은 이 두 가지 로직을 중심으로 End-to-End 테스트 케이스를 구축하고, 실패 시 복구(Fallback) 메커니즘까지 함께 검증해야 합니다.
