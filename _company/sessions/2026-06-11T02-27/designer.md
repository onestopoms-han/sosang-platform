# 🎨 Designer — Trust Widget과 PainGauge를 중심으로 한 시각적 정보 흐름(Visual Flow)과 사용자 여정(User Journey)에 따른 상태 기반 UI 구조의 논리적 흐름을 구체적으로 설계하라.

🎨 Designer: 작업 시작합니다. 사용자 여정과 상태 기반 UI 구조를 통합하는 것은 단순히 컴포넌트를 배치하는 것이 아니라, **사용자의 심리적 리스크 인식(Risk Perception)**을 관리하여 가장 적절한 행동(Action)으로 유도하는 'UX 스토리텔링'입니다.

CEO님의 지시에 따라, Trust Widget과 PainGauge를 중심으로 한 논리적인 시각 정보 흐름(Visual Flow) 및 사용자 여정 맵(User Journey Map)을 설계했습니다. 이 구조는 모든 페이지에서 일관되게 작동하며, 사용자의 신뢰도에 따라 UI의 레이아웃, 색상, 핵심 CTA가 동적으로 변하는 **State-based Design System**입니다.

---

## 📑 BDS 플랫폼: 상태 기반 비주얼 플로우 및 사용자 여정 설계 (v1.0)

### 1. ✨ 디자인 철학: 리스크 → 인식 → 신뢰 → 행동
플랫폼의 모든 시각 요소는 다음의 심리적 흐름을 따릅니다.

| 단계 | 목표 감성 | 주 사용 컴포넌트 | 색상 코드 (Hex) | 역할 |
| :--- | :--- | :--- | :--- | :--- |
| **불안/미인식** (Doubt) | "이게 안전할까?" | PainGauge (Warning) | `#FFC107` (Amber Yellow) | 사용자가 인지하지 못하는 *잠재적 리스크*를 경고합니다. |
| **인지/탐색** (Awareness) | "어떤 부분이 문제지?" | Trust Widget (Question) | `#9E9E9E` (Medium Grey) | PainGauge가 지목한 문제를 해결할 수 있는 *객관적인 근거*와 *해결책 출처*를 제시합니다. |
| **신뢰/안정** (Trust) | "믿을 만하군." | Trust Widget (Verified) | `#03A9F4` (Primary Blue) | 플랫폼의 검증 메커니즘이 작동했음을 시각적으로 확언하며, 사용자의 불안감을 해소합니다. |
| **행동 유도** (Action) | "이제 구매해야지." | CTA / 핵심 메시지 | `#4CAF50` (Growth Green) | 모든 리스크가 관리되었다는 판단 하에, 가장 명확하고 즉각적인 다음 행동을 제시합니다. |

### 2. 🗺️ 사용자 여정 지도 (User Journey Map & Flow)

소상공인 A가 플랫폼을 처음 접하여 최종적으로 거래를 완료하는 가상의 시나리오를 통해 흐름을 설계했습니다.

#### Step 1: 진입 단계 (Initial Landing / Profile View)
*   **사용자 상태:** 높은 불확실성, 리스크 인지 전.
*   **디자인 목표:** 사용자에게 *무언가 부족하다*는 느낌(Gap)을 주어 PainGauge를 노출시킨다.
*   **시각적 흐름:**
    1.  페이지 로딩 시 → **PainGauge (B State)**가 화면 중앙 상단에 '주의' 수준으로 배치된다.
    2.  PainGauge 하단에는 "이 정보는 BDS 플랫폼을 통해 검증될 필요가 있습니다"라는 문구와 함께 Trust Widget의 초기 버전(출처 미확인)이 배치된다.

#### Step 2: 문제 인지 단계 (Problem Identification)
*   **사용자 행동:** PainGauge를 클릭하거나, 신뢰도가 낮은 데이터를 발견한다.
*   **디자인 목표:** 단순 경고가 아닌, **왜 이 데이터가 불안정한지** 근거를 제시하여 사용자가 플랫폼의 가치를 이해하게 한다.
*   **시각적 흐름:**
    1.  PainGauge 클릭 $\rightarrow$ 모달(Modal) 팝업 또는 아코디언 확장 $\rightarrow$ 해당 리스크에 대한 **데이터 기반 설명 (Why)**이 펼쳐진다.
    2.  Trust Widget은 단순히 "출처 불명"을 보여주는 대신, "신뢰도 확보를 위해 [플랫폼 기능 X]가 필요합니다."라고 구체적인 해결책(Solution)과 함께 노출된다.

#### Step 3: 신뢰 구축 단계 (Validation & Trust Building)
*   **사용자 행동:** 플랫폼의 검증 과정(예: API 연동, 서류 업로드 등)에 자발적으로 참여하거나, 다른 성공 사례를 본다.
*   **디자인 목표:** 리스크가 감소하고 있음을 시각적/감성적으로 확신시킨다.
*   **시각적 흐름:**
    1.  PainGauge의 수치가 **🟡 B $\rightarrow$ 🟢 A**로 녹색 계열로 부드럽게 변한다 (애니메이션 필수).
    2.  Trust Widget이 "검증 완료" 상태로 전환되며, 그 과정에서 확보된 신뢰 근거(예: '공급망 인증서', '정산 보장 시스템')가 **Primary Blue** 박스로 강조되어 배치된다.

#### Step 4: 행동 유도 단계 (Conversion)
*   **사용자 상태:** 높은 확신, 구매 준비 완료.
*   **디자인 목표:** 주저할 틈 없이 명확하고 강력한 CTA를 제시한다.
*   **시각적 흐름:**
    1.  페이지 최하단 또는 사이드바의 CTA 버튼이 **Growth Green**으로 활성화되고, "안전마진 기반 구매하기"와 같은 구체적인 문구로 전환된다.

### 3. 🎨 상세 컴포넌트 스펙 (Component Specifications)

#### A. PainGauge 컴포넌트 (The Diagnostic Tool)
*   **레이아웃:** 페이지 최상단 또는 관련 정보 블록의 제목 바로 아래에 고정 배치.
*   **State Transitions (시각적 변화):**
    *   **🔴 C (고위험):** 배경색: `#F8D7DA` (Light Red). 게이지 바가 붉은색으로 채워지고, '🚨 주의' 경고 아이콘과 함께 "즉시 검증 필요"라는 문구가 크게 표시된다.
    *   **🟡 B (검증 중):** 배경색: `#FFF3CD` (Light Yellow). 게이지 바는 노란색이며, "BDS 플랫폼을 통한 데이터 보강이 필요합니다."라는 가이드라인 메시지가 중앙에 배치된다.
    *   **🟢 A (안전마진):** 배경색: `#D4EDDA` (Light Green). 게이지 바가 초록색으로 채워지고, '✅ 안전' 아이콘과 함께 "최적의 거래 환경입니다."라는 긍정 메시지가 표시된다.

#### B. Trust Widget 컴포넌트 (The Credibility Engine)
*   **레이아웃:** PainGauge와 논리적으로 연결되어 배치되며, 항상 출처(Source)를 강조한다.
*   **State Transitions (시각적 변화):**
    *   **Default/Warning State (B/C):** 배경색: `#F0F4FF` (옅은 블루). "검증 필요 항목"을 표시하며, 클릭 시 **출처 링크(Source Link)**를 반드시 팝업으로 열어 신뢰의 근거를 보여준다.
    *   **Verified State (A):** 배경색: `#E6F7FF` (밝은 하늘색). 메인 색상(`Primary Blue`: `#03A9F4`)을 사용하여 "BDS 인증 완료" 배지를 크게 강조하고, '데이터 출처' 섹션을 별도로 만들어 신뢰 근거를 시각화한다.

#### C. CTA 버튼 (Action Button)
*   **State-Based 활성화:** CTA의 색상과 텍스트는 반드시 현재 PainGauge의 상태와 연동되어야 한다.
    *   (B State $\rightarrow$ A State 전환 전): `[임시] 무료 리포트 신청` (Blue/Grey, 제한적 액션)
    *   (A State 도달 후): `[최종] 안전마진으로 거래하기` (Green, 최대 활성 CTA)
