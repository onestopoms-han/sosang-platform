🎨 Designer: 작업 시작합니다.

CEO의 지시에 따라, 지금까지 정의된 복잡한 컴포넌트(Trust Widget 및 PainGauge)와 전반적인 사용자 흐름을 개발자 코다리에게 최종적으로 인계하고 시스템 안정성을 확보하기 위한 **통합 검토 체크리스트**를 작성하겠습니다.

이 문서는 단순한 디자인 가이드가 아니라, **'이대로 구현되면 비즈니스 목표(안전마진 수익화) 달성 여부'**와 **'디자인 시스템 준수 여부'** 두 가지 관점에서 모든 팀원이 최종적으로 검토할 수 있는 싱글 트루스 소스(Single Source of Truth)가 되어야 합니다.

체크리스트는 다음의 파일로 저장하고, 향후 모든 에이전트들이 개발/테스팅 시 참고하도록 공유하겠습니다.

## 📝 목적
본 문서는 Trust Widget 및 PainGauge 컴포넌트를 포함한 핵심 기능의 디자인-개발 이관(Handoff) 단계에서, 시각적 일관성(Visual Consistency), 사용자 경험 흐름(UX Flow), 비즈니스 로직 구현 적합성을 최종 검증하기 위해 작성되었습니다. 모든 에이전트가 본 체크리스트를 통해 규격 준수 여부를 확인해야 합니다.

## 🔍 섹션 1: 디자인 시스템 및 시각적 일관성 (🎨 Designer 관점)
**목표:** 브랜드 가이드라인(색상, 타이포그래피, 간격 등)을 벗어나는 모든 요소가 없는지 검증합니다.

| 체크 항목 | 규정/요구사항 | 점검 내용 | 준수 여부 (Y/N) | 비고 (문제 발생 시 수정 필요 사항) |
| :--- | :--- | :--- | :--- | :--- |
| **컬러 팔레트** | Primary: Deep Blue (#004D66), Secondary: Growth Green (#3CB371). Accent/Warning: [지정된 경고색] | 1. 모든 배경, 강조 요소가 지정 컬러 코드를 사용했는가? (예: CTA 버튼의 색상) | [ ] | |
| **타이포그래피** | 주력 서체(Font Family) 및 계층 구조(H1-Body). | 1. 제목/본문/캡션 등 모든 텍스트에 지정된 폰트와 크기가 적용되었는가? (특히 작은 경고 문구) | [ ] | |
| **여백 및 그리드** | 최소 간격(Spacing Unit)은 8px의 배수를 준수해야 함. | 1. 컴포넌트 내부 요소 간, 섹션 간 여백이 일관된 Grid System을 따르는가? (좌우/상하 정렬 포함) | [ ] | |
| **아이콘 시스템** | BDS Component Library에 정의된 아이콘 세트를 사용해야 함. | 1. 임의로 제작되거나 통일성이 부족한 아이콘이 사용되지 않았는가? (예: 화살표 방향, 스타일 일관성) | [ ] | |

## ⚙️ 섹션 2: 기능적 흐름 및 UX 로직 검증 (🧑‍💻 개발/테스터 관점)
**목표:** 사용자가 컴포넌트를 사용하는 모든 단계(State Transition)가 비즈니스 목표에 맞게 자연스럽고 명확한지 검증합니다.

| 체크 항목 | 상세 요구사항 | 테스트 시나리오 | 예상 결과값 (Success State) | 준수 여부 (Y/N) |
| :--- | :--- | :--- | :--- | :--- |
| **PainGauge 변화** | 1. 데이터 입력 및 애니메이션 트리거 로직의 정확성. | 사용자 A가 '만족도 저하' 데이터를 입력했을 때, PainGauge 수치가 즉시 하락하며 경고 색상(Warning Color)이 활성화되는가? | 시각적 피드백: (빨강 계열 톤 변화), 데이터 로깅 API 호출 성공. | [ ] |
| **Trust Widget 상호작용** | 2. 신뢰도 증가 메커니즘의 직관성. | 사용자가 '컨설팅 신청'을 완료했을 때, Trust Widget 수치가 즉각적으로 상승하며 애니메이션 효과가 발생하는가? | 시각적 피드백: (녹색 계열 톤 변화), 성공 메시지 및 다음 액션(CTA) 제시. | [ ] |
| **에러 핸들링** | 3. 시스템 오류 또는 필수 입력 누락 시 처리 과정. | 사용자가 데이터를 입력하지 않고 제출 버튼을 클릭했을 때, 명확한 에러 메시지와 함께 포커스가 해당 필드로 이동하는가? | 사용자 경험: 친절하고 구체적인 힌트 제공 (개발자용 코드는 숨김). | [ ] |
| **모바일 최적화** | 4. 반응형 디자인 및 접근성(Accessibility). | 태블릿, 모바일 환경에서 모든 컴포넌트가 깨지지 않고, 터치 영역이 충분히 확보되는가? | 가독성/사용성: 글씨 크기 적정 유지, 버튼 누르기 쉬운 간격 확보. | [ ] |

## 💻 섹션 3: 기술 명세 및 개발 구현 검토 (⚙️ 코다리 관점)
**목표:** 디자인 명세를 기반으로 컴포넌트를 실제 코드에 이식할 때 발생하는 기술적 모호성을 제거하고, 구현의 완벽함을 확보합니다.

| 체크 항목 | 기술 사양/규격 | 확인 사항 및 질문 | 최종 확정 값 (Value) | 준수 여부 (Y/N) |
| :--- | :--- | :--- | :--- | :--- |
| **컴포넌트 스코프** | `BDS_Component_Library_V2.0` 기반으로 재사용 가능해야 함. | 1. 신규 컴포넌트를 추가하거나 수정할 때, 반드시 라이브러리의 명명 규칙(Naming Convention)을 따르는가? | (예: `<Button>`, `<GaugeChart>`) | [ ] |
| **API 연동 지점** | 모든 데이터 변화는 백엔드 API 호출에 의해 트리거되어야 함. | 1. PainGauge 수치 변경, Trust Widget 증가 등의 핵심 로직이 *클라이언트 측 변수*가 아닌 *실제 API 응답*에 기반하는가? | `POST /api/gauge-update` 등 (Endpoint 명시) | [ ] |
| **애니메이션 정의** | 모든 트랜지션(Transition)은 시간과 속도가 명확히 정의되어야 함. | 1. 애니메이션의 시작점, 종료점, 지연시간(Delay), 이징 커브(Easing Curve)가 디자인 시스템에 명시되었는가? (예: `ease-out` over 500ms) | CSS/JS 로직 레벨에서 정의된 값 확인. | [ ] |
| **테스트 케이스** | QA 및 테스트 자동화를 위한 Edge Case가 정의되어야 함. | 1. 컴포넌트의 최대치(Max), 최소치(Min), 데이터 누락(Null) 등 극한 상황에 대한 시각적/기능적 처리가 명시되었는가? | (예: Max=100, Min=0, Error State Active) | [ ] |

## 🎯 산출물 설명 및 기대 효과
1.  **체크리스트 통합:** 기존의 파편화된 디자인/개발 요구사항들을 하나의 구조적 문서로 통합했습니다.
2.  **책임 분산:** 각 섹션(디자인, UX 흐름, 기술 명세)을 통해 어느 에이전트가 어떤 관점에서 검증해야 하는지 역할 경계를 명확히 했습니다.
3.  **개발 준비 완료:** 코다리에게 전달할 때 필요한 '무엇'과 '어떻게 테스트 할 것인가'에 대한 가이드라인을 제공함으로써, 개발 단계의 불필요한 Q&A와 재작업 위험(Rework Risk)을 최소화합니다.
