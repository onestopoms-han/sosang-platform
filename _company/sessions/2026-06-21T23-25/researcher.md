## 📌 1. 배경 및 목적
- **배경:** 현재 정의된 `Trust Widget` 및 `PainGauge` 의 디자인 명세와 API 스키마 간에 존재하는 '기술적 난제(구현 격차)'를 해결하기 위해 실행 가능한 로드맵을 수립합니다.  
- **목표:** 개발팀과 디자인팀이 즉시 협업하여 MVP 핵심 대시보드를 구현할 수 있도록 명확한 우선순위 액션 아이템을 제시합니다.  
- **핵심 원칙:**  
  - **MVP 집중:** '신뢰도 표시'와 '위험 지표' 시각화에 모든 자원을 투입합니다.  
  - **디자인/개발 동기화:** 디자인 명세 (Figma) 와 실제 코드 구조를 매핑하여 격차를 최소화합니다.

## 🛠️ 2. 기술적 난제 식별 및 해결 전략
### 🔴 현재 존재하는 주요 난제
| 난제 유형 | 설명 | 해결 방안 |
|-----------|------|----------|
| **상태 정의 불일치** | 디자인 명세 (Critical/Warning) 와 API 응답 스키마 값 (`status: "high_risk"`) 간 매핑 규칙 미정확 | API 스키마와 상태 텍스트 매핑 테이블을 Phase 1 에서 확정합니다. |
| **동적 색상 연동 부재** | 데이터 상태 변화 시 UI 색상 변경 로직 (CSS 변수/조건부 렌더링) 정의 누락 | Phase 2 에 CSS 변수를 활용하여 상태별 컬러 톤을 컴포넌트 프로퍼티로 노출합니다. |
| **에러 핸들링 미정의** | API 파싱 실패 시 UI fallback 동작 (예: 로딩 중 표시, 기본 색상 적용) 규칙 없음 | Phase 3 에서 에러 상태 시각화 및 UI fallback 로직을 구현합니다. |

## 🗺️ 3. 우선순위 로드맵 (Next Steps)
각 단계는 **개발팀**과 **디자인팀**이 협업하여 즉시 실행할 수 있도록 설계되었습니다.

### **Phase 1: 기초 명세 정합성 확보 (우선순위: 🔴)**  
> **기간:** T+3 일 이내 완료  
> **주요 액션:** API 스키마 확정 및 상태 매핑 규칙 문서화  

- **[개발팀]** `Dashboard_API_Response_Schema_v1.0.md` 파일을 기반으로 상태값 (`status`) 과 시각적 의미 (예: `"critical"`, `"warning"`, `"normal"`) 간의 일대일 매핑 테이블 생성  
  ```markdown
  | API Status Key | 시각화 상태 (UI) | 색상 코드 (Hex) | 텍스트 라벨 |
  |----------------|------------------|-----------------|-------------|
  | high_risk      | Critical          | #FF4D4F          | "위험"      |
  | medium_risk    | Warning           | #FF9C3E          | "주의"      |
  | low_risk       | Normal            | #52C41A          | "정상"      |
  ```  
- **[디자인팀]** 위 테이블을 Figma 라이브러리의 컴포넌트 속성 (Properties) 에 반영하여 개발자가 쉽게 참조할 수 있도록 합니다.  
- **[공동 작업]** API 응답 구조를 변경할 경우, 매핑 규칙도 동시 업데이트되도록 협업 프로세스 정립  

### **Phase 2: 컴포넌트 구현 및 초기 연동 (우선순위: 🟠)**  
> **기간:** T+7 일 이내 완료  
> **주요 액션:** 기본 렌더링 로직 및 상태 기반 색상 적용 구현  

- **[개발팀]** `Trust Widget` 과 `PainGauge` 컴포넌트 (`trust-widget.tsx`, `pain-gauge.tsx`) 에 Phase 1 의 매핑 테이블을 적용하여 기본 렌더링 로직 구현  
  - **반응형 디자인:** CSS Grid 를 활용하여 레이아웃 최적화 및 모바일 대응 (Flexbox fallback)  
- **[디자인팀]** Figma 프로토타입의 컴포넌트 변형 (Variants) 을 위 테이블과 동기화하여 실제 렌더링 결과와 시각적 일관성 검증  
  - **검증 기준:** 위/아래 페이지 전환 시 색상 변화가 부드럽게 적용되는지 확인  

### **Phase 3: 동적 반응성 및 UX 최적화 (우선순위: 🟡)**  
> **기간:** T+14 일 이내 완료  
> **주요 액션:** 데이터 상태 변경에 따른 애니메이션 및 에러 처리 로직 구현  

- **[개발팀]** 상태 값 (`status`) 이 변경될 때 CSS 트랜지션 (`transition: background-color 0.3s ease`) 을 적용하여 색상 전환을 부드럽게 만듦  
  - **에러 핸들링:** API 응답 지연 시 `Loading` 스피너 표시, 오류 발생 시 기본 `Normal` 색상으로 fallback  
- **[디자인팀]** 애니메이션 타이밍 (0.3s) 과 색상 변화 패턴이 사용자 경험을 저해하지는 않는지 UX 테스트용 시나리오 작성  
  - **검증 기준:** 데이터 업데이트 빈도에 따른 UI 부하 및 색상 깜빡임 최소화  

### **Phase 4: 통합 검증 및 시그 오프 준비 (우선순위: 🟢)**  
> **기간:** T+21 일 이내 완료  
> **주요 액션:** 모의 데이터 테스트, 최종 디자인/개발 리뷰  

- **[개발팀]** 위상진행 (Mock Data) 을 기반으로 컴포넌트 전체를 연동하여 기능 검증 (단일 컴포넌트 → 전체 대시보드)  
  - **테스트 시나리오:** 정상 상태, 위험 상태, 데이터 로딩 중, API 에러 등 모든 상태에 대한 UI 반응 확인  
- **[디자인팀]** 최종 Figma 프로토타입과 실제 렌더링 결과 비교 분석하여 미준수 항목 수정  
  - **검증 기준:** 디자인 명세 (Design Spec) 와 구현 결과물의 시각적/기능적 일치도 95% 이상
