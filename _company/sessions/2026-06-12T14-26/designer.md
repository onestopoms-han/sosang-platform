### 🚨 TrustWidget/PainGauge 베타 QA 보고서 (V1.1 수정 제안)

**점검 환경 가정:**
*   베타 사용자는 디지털 기기 활용도가 낮은 소상공인일 가능성이 높음. (직관성 최우선)
*   네트워크 불안정 및 저화질 모니터 접근 빈도 높음.
*   데이터 입력/표시가 비전문적(임의 수치, 감성적 설명 등)할 수 있음.

#### 1. 인터랙션/사용자 흐름 리스크 (Interaction & Flow)

| 영역 | 잠재적 오류/리스크 지점 | 수정 제안 (Designer Action) | 관련 스펙 위치 |
| :--- | :--- | :--- | :--- |
| **PainGauge 초기 진입** | 사용자가 데이터를 직접 입력할 때, 수치에 대한 의미를 모르면 혼란을 겪고 이탈함. ('왜 이 숫자를 넣어야 하는가?') | 데이터 입력 필드 위에 '간단 가이드(Tooltip)'를 반드시 추가하고, 예시 값과 함께 직관적인 설명('예: 매출 대비 비용 비율')을 제공해야 함. | PainGauge Input Flow |
| **성장 로드맵 전환** | `TrustWidget`의 분석 결과가 다음 단계(`Growth Plan`)로 넘어갈 때, 시각적/심리적 연결고리가 약해짐. (단순히 다른 화면으로 이동) | **'Transition Animation' 추가:** 진단(PainGauge) $\rightarrow$ 인식('이것이 문제다') $\rightarrow$ 해결책 제시(TrustWidget 분석 결과) $\rightarrow$ 행동 촉구(Next Step 버튼). 단계별로 부드러운 시각적 흐름을 만들어야 함. | 전체 여정 지도 (Journey Map) |
| **상세 설명 팝오버** | 특정 지표를 클릭했을 때 나오는 상세 설명(`Tooltip` 또는 `Modal`)이 너무 크거나 복잡하면, 사용자가 핵심 메시지(안전마진)에서 벗어나 길을 잃음. | 모달/툴팁의 내용은 반드시 **한 문장 요약 + 한 가지 행동 제안**으로 축소해야 함. 정보 과부하 방지를 위해 레이어 깊이를 최소화할 것. | 컴포넌트 상세 설명 (Detail Modal) |

#### 2. 시각적 일관성 리스크 (Visual Consistency & UI/UX)

| 영역 | 잠재적 오류/리스크 지점 | 수정 제안 (Designer Action) | 관련 스펙 위치 |
| :--- | :--- | :--- | :--- |
| **컬러 대비(Contrast)** | 텍스트와 배경색의 명도 차이가 낮은 경우, 특히 고령층이나 눈이 피로한 사용자가 중요한 수치나 CTA를 인지하지 못함. | 모든 중요 텍스트와 CTA 버튼은 WCAG AA 레벨 이상의 높은 대비율을 유지하도록 강제하고, 시스템 색상 외에 **강조용 블랙/화이트 조합**을 추가하여 안정성을 확보할 것. | Typography & Color System |
| **데이터 시각화 복잡도** | `TrustWidget`의 그래프나 차트가 너무 많은 요소(축, 범례, 라벨)를 포함하면, 사용자가 '무엇이 중요한지' 즉시 파악하기 어려움. | 초기 로드 상태에서는 **가장 핵심적인 트렌드만 강조**하는 미니멀한 디자인을 기본값으로 설정하고, 상세 분석 페이지에서만 복잡도를 높이는 2단계 방식을 채택할 것. (Default State vs Deep Dive State) | 그래프/차트 컴포넌트 |
| **반응형 레이아웃 실패** | 태블릿이나 스마트폰 등 작은 화면에서 TrustWidget의 다중 차트가 수평으로 펼쳐지면, 좌우 스크롤만으로는 전체 맥락 이해에 어려움이 발생함. | 모든 데이터 시각화 컴포넌트는 반드시 '카드 기반(Card-based)' 레이아웃을 기본 원칙으로 삼고, 화면 크기에 따라 **수직 적층(Vertical Stacking)**되는 것을 최우선으로 고려해야 함. | Responsive Layout Principle |

#### 3. 에지 케이스/데이터 리스크 (Edge Case Handling)

| 영역 | 잠재적 오류/리스크 지점 | 수정 제안 (Designer Action) | 관련 스펙 위치 |
| :--- | :--- | :--- | :--- |
| **'데이터 없음(No Data)' 상태** | 시스템이 데이터를 불러오지 못하거나, 사용자가 아무것도 입력하지 않은 'Empty State'가 단지 하얀 공간으로 처리될 경우, 사용자에게 무력감과 불신을 줌. | 빈 상태일지라도 **다음 액션(Next Action)에 대한 가이드**를 제공해야 함. 예: "데이터가 없어 불안합니다. 다음 주 매출 데이터를 입력해보세요!" 와 같이 심리적 공감을 포함한 메시지를 디자인 요소로 활용할 것. | Empty State Design |
| **시스템 오류 발생 시** | API 호출 실패, 네트워크 연결 끊김 등의 시스템 오류가 사용자에게 단순한 '에러 코드(Error Code)' 형태로 노출될 경우. | 기술적인 언어 대신, 사용자의 관점에서 이해 가능한 **'감성적 안내 메시지'**를 제공해야 함. 예: "잠시 인터넷 연결이 불안정합니다. 🔄 다시 시도하거나 담당자에게 문의해주세요." (안전마진 컨셉 적용) | Error Handling UI/UX |
