# 🎯 BDS 소상공인플렛폼 — Pilot 프로그램 경험 흐름 최종 검증 명세서 (V2.0)

**작성 목적:** Writer의 가치 제안 메시지(Pain Point 강조)와 Developer가 정의한 추적 로직을 통합하여, 사용자 온보딩부터 서비스 이용까지 모든 인터랙션 지점의 정보 구조 및 시각적 흐름(UX/UI) 일관성을 검증한다.
**검토 범위:** Pilot 신청서 작성 → 계정 생성 → Dashboard 진입 (최초 30분 경험)

## I. 🚀 Phase 1: Pain Point 유도 및 참여 동의 (Onboarding / `user_registered` Trigger)
*   **목표 메시지:** "당신은 지금 데이터 기반 성장이 필요한 위험 상태에 있습니다."
*   **주요 화면:** Pilot 신청서 작성 페이지 (Writer 초안 기반 재구성)

| 검증 항목 | 내용 및 요구사항 | 체크리스트 (Y/N) | 담당 에이전트 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **[A1] 위기감 조성** | 신청서 상단에 PainGauge 기반의 진단 결과를 '사전에' 노출해야 함. (예: "매출 데이터 분석 필요", "리스크 경고 레벨 3") | [ ] | Designer/Writer | *최초 시각적 충격점* |
| **[A2] 정보 구조 적합성** | 신청서 질문 항목이 단순한 '정보 수집' 목적을 넘어, **'자체 문제 진단'을 유도하는 문항**으로 재구성되어야 함. (예: "어떤 데이터가 부족하다고 느끼시나요?") | [ ] | Writer | 기술적 제약 고려 필수 |
| **[A3] 추적 로직 연결** | 신청서 제출 완료 직후, `user_registered` 이벤트 트리거와 함께 가입 목적 및 진단 결과(PainGauge 점수)를 백엔드에 전송하는 플로우가 구현되어야 함. | [ ] | Developer | 기술 검증 필수 |
| **[A4] CTA 명확성** | 다음 단계로의 이동 버튼(CTA)이 '신청 완료'가 아닌, **'AI 진단 리포트 받기 (다음 단계)'**와 같이 기대감을 심어주는 문구여야 함. | [ ] | Designer | 감성적 동기 부여 |

## II. 🖥️ Phase 2: 핵심 가치 경험 및 데이터 학습 (Dashboard Access / `dashboard_view` Trigger)
*   **목표 메시지:** "우리는 당신의 문제를 해결할 도구와 로드맵을 가지고 있습니다."
*   **주요 화면:** 첫 진입 대시보드 (Welcome Screen)

| 검증 항목 | 내용 및 요구사항 | 체크리스트 (Y/N) | 담당 에이전트 | 비고 |
| :--- | :--- | :--- | :--- | :--- |
| **[B1] 즉각적 가치 제시** | 대시보드 진입 시, 신청서에서 발견된 '가장 큰 문제'를 해결하는 로드맵(Action Plan)을 가장 크게 보여주어야 함. (정보의 위계화) | [ ] | Designer | *최우선 배치 영역* |
