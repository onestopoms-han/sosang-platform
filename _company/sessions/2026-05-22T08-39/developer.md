# 📄 BDS 소상공인 플랫폼: 진단 API 명세서 (v1.0)

**목표:** `DiagnosisInput` 및 `DiagnosisResult` 계약을 기반으로, 데이터 무결성을 보장하며 안정적으로 작동하는 백엔드 API 구현을 시작한다.

## 1. 전제 조건 및 원칙 (Stability First)

1.  **데이터 무결성(Data Integrity):** 모든 입력 데이터(`DiagnosisInput`)는 스키마 검증을 통과해야 하며, 처리 과정에서 데이터 손실이나 변형이 발생해서는 안 된다.
2.  **에러 처리(Error Handling):** API 호출 실패, 유효성 검사 실패, 데이터베이스 오류 등 모든 잠재적 실패 지점에 대해 명확하고 구체적인 에러 코드와 메시지를 반환해야 한다.
3.  **데이터 흐름 분리:** 진단 로직(`diagnosis_service.py`)의 비즈니스 로직과 데이터 입출력(I/O) 로직을 명확히 분리하여, 추후 확장 시 모듈 간 결합도를 낮춘다.

## 2. 입력 API 명세: `DiagnosisInput` (POST /api/v1/diagnosis)

**목표:** 소상공인의 상황에 대한 진단 요청 및 필요한 모든 컨텍스트 데이터를 수집한다.

### 2.1. 엔드포인트 상세
*   **메소드:** `POST`
*   **경로:** `/api/v1/diagnosis`
*   **인증:** 필수 (JWT 또는 API Key 기반)

### 2.2. 요청 본문 스키마 (`DiagnosisInput`)

| 필드명 | 타입 | 필수 여부 | 설명 | 제약 조건 및 유효성 검사 | 근거 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `business_type` | String | 필수 | 소상공인의 사업 유형 (예: 음식점, 소매업, 서비스업) | Enum 값으로 제한. 허용된 값 리스트 확인. | [코다리 개인 메모리] |
| `pain_point_score` | Float | 필수 | 현재 겪고 있는 주요 고통 지표 점수 (0.0 ~ 10.0) | $0 \le x \le 10$ 범위 검증. | [코다리 개인 메모리] |
| `financial_risk_data` | Object | 필수 | 재정적 위험 데이터 객체 | 내부 `ROI_DataModel` 스키마 준수 (자세한 명세는 별도 참조). | [코다리 개인 메모리] |
| `contextual_factors` | Array of Object | 선택 | 상황을 둘러싼 추가 맥락 정보 리스트 | 각 객체는 최소 필드(예: `factor_name`, `description`)를 포함해야 함. | [코다리 개인 메모리] |
| `user_id` | String | 필수 | 진단 요청 사용자 ID (인증된 사용자 매핑) | DB 외래 키 검사 필요. | [추측] |

## 3. 출력 API 명세: `DiagnosisResult` (Response Body)

**목표:** 입력된 데이터를 기반으로 정량적 분석 결과와 행동 계획을 제공한다.

### 3.1. 응답 구조 상세
*   **상태 코드:** HTTP Status Code는 성공 시 `200 OK`, 실패 시 적절한 클라이언트 에러 코드(4xx, 5xx)를 반환한다.

### 3.2. 응답 본문 스키마 (`DiagnosisResult`)

| 필드명 | 타입 | 필수 여부 | 설명 | 데이터 흐름 안정성 고려사항 | 근거 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `diagnosis_id` | String | 필수 | 본 진단 요청의 고유 ID. (DB PK 매핑) | 데이터 무결성을 위해 반드시 생성되어야 함. | [추측] |
| `risk_level` | String | 필수 | 종합 위험 수준 (예: Critical, High, Medium, Low). **빨간색 톤 강조.** | `financial_risk_data` 기반으로 결정하며, 시각적 연관성 확보. | [회사 공동 목표/최근 의사결정] |
| `estimated_loss` | Float | 필수 | 현재 상황에서 예상되는 잠재 손실 비용 (손실 비용 프레임 적용). | ROI 계산 로직(`roiCalculator.ts`)의 결과를 기반으로 산출. | [코다리 개인 메모리] |
| `action_plan` | Object | 필수 | 구체적인 행동 계획 목록. | 각 단계는 명확한 실행 가능한 액션 항목으로 구성되어야 함. | [Action Plan 기능 구현 지시] |
| `recommendation` | String | 필수 | 핵심 요약 및 즉각적인 행동 촉구 메시지 (손실 비용 강조). | 마케팅 카피 원칙(`손실 = 보험료`)을 반영하여 작성. | [회사 공동 목표/최근 의사결정] |
| `data_metrics` | Object | 선택 | 세부 데이터 지표 및 시각화에 필요한 데이터. | 프론트엔드 차트 구현을 위해 정량적 데이터를 제공. | [코다리 개인 메모리] |

## 4. 데이터 흐름 안정성 확보 방안 (Stability Protocol)

1.  **프론트-백엔드 계약 강제:** 프론트엔드 컴포넌트는 반드시 `DiagnosisResult`의 스키마를 따르도록 강제한다. 만약 백엔드가 이 스키마를 벗어나는 응답을 반환하면, 즉시 400 Bad Request 에러를 발생시킨다.
2.  **비즈니스 로직 격리:** 핵심 계산 로직(`roiCalculator.ts`의 결과)은 별도의 서비스 레이어를 통해 처리하고, API 엔드포인트는 이 서비스 레이어에만 의존하도록 설계한다. 이는 데이터 변형이 전체 시스템에 미치는 영향을 최소화한다.
3.  **Negative Testing 통합:** 모든 입력 데이터(특히 `financial_risk_data`)에 대해 사전에 정의된 Negative Testing 시나리오(`sessions/2026-05-21T05-12/developer.md` 등)를 백엔드 레벨에서 검증하는 로직을 추가한다.
4.  **트랜잭션 관리:** 데이터베이스 쓰기 작업(진단 저장, 결과 기록)은 원자성(Atomicity)을 보장하기 위해 트랜잭션 내에서 처리되어야 한다.
