# 🚀 MVP 와이어프레임 테스트 스크립트 통합 마일스톤 및 작업 분배 계획 (V1.0)

**작성자:** 영숙 (Personal Assistant)
**목표:** 소상공인 플랫폼의 핵심 사용자 여정(Pain $\rightarrow$ Solution)을 담은, 클릭 가능한 프로토타입 완성.
**기준 시점:** 2026년 6월 28일

---

## 🎯 핵심 과제 요약: '흐름'과 '데이터' 연결 고리 만들기
현재 가장 큰 병목 지점(Bottleneck)은 **디자인 의도를 기술적 흐름으로 변환**하고, 특히 Codari가 지적한 **실시간 데이터 스트리밍 기능 구현**입니다. 이 부분을 최우선 과제로 잡고 작업을 분배해야 합니다.

## 🗓️ 통합 마일스톤 (3단계 액션 플랜)

| 단계 | 목표 산출물 | 핵심 내용 | 예상 소요 시간 |
| :--- | :--- | :--- | :--- |
| **Phase 1: Blueprint 확정** | `StoryFlowSchema_Final.v2.0` | 사용자 여정(PainGauge $\rightarrow$ Trust Widget)의 API 호출 순서와 상태 전이 규칙을 최종 정의합니다. (가장 중요!) | 1일 |
| **Phase 2: Tech Skeleton 구축** | `Wireframe_BaseCode.v1.0` | Codari가 제안한 대로, 실시간 데이터 수신 및 상태 변화를 처리할 백엔드/프론트엔드 기본 구조(뼈대)를 완성합니다. | 3일 |
| **Phase 3: Content & Interactivity 구현** | `Testable_Prototype_v1.0` | 정의된 스크립트를 따라, 실제 콘텐츠(케이스 스터디, 솔루션 카드)와 인터랙티브 요소를 채워 넣고 테스트 가능한 프로토타입을 완성합니다. | 2일 |

## 🧑‍💻 에이전트별 역할 분배 및 액션 아이템 (Action Item List)

### 🥇 [최우선] 코다리 (Backend/Architecture Lead)
*   **역할**: 시스템의 기술적 구조 설계 및 데이터 흐름 정의.
*   **주요 과제**:
    1.  `StoryFlowSchema_Final.v2.0` 작성: Researcher와 Designer가 제시한 모든 상호작용(Red $\rightarrow$ Yellow $\rightarrow$ Green)을 커버하는 **상태 전이 규칙 (State Transition Rules)**과 필요한 API 엔드포인트 목록을 완성합니다.
    2.  실시간 데이터 스트리밍 로직 정의: WebSocket 기반의 실시간 지표 업데이트 및 처리를 위한 백엔드 구조를 설계하고, 초기 코드를 작성합니다.

### 🥈 [협력] Designer (UX/UI Lead)
*   **역할**: 기술적 제약에 맞는 최종 인터랙션 플로우 확정.
*   **주요 과제**:
    1.  `Interaction_Flowchart_v2.0`: 코다리에게 정의된 API 스키마(V3.0)를 기반으로, **'클릭했을 때 실제로 어떤 컴포넌트가 어떤 데이터로 바뀌어 보여야 하는지'**에 대한 구체적인 플로우차트를 제공합니다. (와이어프레임의 '동작 시나리오' 역할).
    2.  최종 디자인 핸드오프: 팝업/모달 형태의 증거 데이터(Case Study) 영역의 인터랙션 디테일을 확정하여 개발팀에 전달합니다.

### 🥉 [지원] Researcher (Content/Journey Lead)
*   **역할**: 흐름을 뒷받침하는 콘텐츠와 사용 시나리오 검증.
*   **주요 과제**:
    1.  데이터 증거 확보: PainGauge가 Red 상태일 때, 사용자에게 가장 먼저 보여줘야 할 **'최소한의 행동 가이드라인(Minimum Action Guideline)'** 3가지 핵심 메시지를 확정하여 콘텐츠팀에 전달합니다. (빈 화면 방지)

### ✨ [검증] 모든 에이전트
*   **역할**: 각자 맡은 결과물을 종합적으로 테스트하고, '사용자가 가장 당황할 만한 지점'을 찾아내어 개선점을 보고합니다.
