# 💡 UI 컴포넌트 최종 State 검증 및 미세 조정 가이드라인 (Loss Gauge & Red/Green System)

**대상:** 프론트엔드 개발팀
**목표:** Loss Gauge 시스템의 시각적 명확성과 사용자 경험(UX)을 극대화하여, 소상공인의 리스크 인지 및 행동 변화를 유도하는 가장 매력적인 UI 상태를 정의한다.

## 1. 핵심 색상 및 의미론 (Color & Semantics)

| 시스템 | 색상 코드 (HEX) | 역할/의미 | 적용 위치 |
| :--- | :--- | :--- | :--- |
| **기본/안정 (Base/Safe)** | `#004D66` (Deep Blue) | 시스템의 기본 상태, 안전 범위, 잠재적 기회 | 배경, 컨테이너 프레임, 기본 텍스트 |
| **위험 경고 (Warning/Risk)** | `#FFC107` (Amber/Yellow) | 주의 단계, 리스크가 증가하고 있음을 알림 | Loss Gauge 중간 레벨 |
| **심각한 위험 (Critical Risk)** | `#D32F2F` (Red) | 즉각적인 행동이 필요한 심각한 손실 경고 | Loss Gauge 상위 레벨 (경고선), 최종 실패 상태 |
| **성장/해결 (Growth/Solution)** | `#3CB371` (Growth Green) | 긍정적 변화, 안전 목표 달성, 생존 전략 제시 | Loss Gauge 하위 레벨 (안전 범위), 성공 지표 |

## 2. Loss Gauge 상태별 UI State 정의 (State Mapping)

Loss Gauge 값(손실액 또는 리스크 지수)에 따라 컴포넌트가 보여줘야 할 명확한 시각적 상태를 정의합니다. 모든 변화는 **부드러운 전환(Smooth Transition)**을 통해 이루어져야 합니다.

| Loss Gauge 값 범위 | 시각적 상태 (Visual State) | 레이블/텍스트 가이드 | 액션 유도 (CTA Implication) |
| :--- | :--- | :--- | :--- |
| **0% ~ 25%** | **Safe Zone** (Deep Blue 배경) | "안정", "기회 영역" | 모니터링 및 학습 권장 |
| **26% ~ 50%** | **Warning Zone** (Amber 경고) | "주의 필요", "경계 시작" | 즉각적인 분석 및 전략 검토 유도 |
| **51% ~ 75%** | **High Risk Zone** (Orange/Red 그라데이션 시작) | "위험 증가", "생존 모드 진입" | 구체적인 리스크 회피 전략 제시 |
| **76% ~ 100%** | **Critical Zone** (Strong Red) | "심각한 손실 임박", "긴급 대응 필요" | 즉각적인 컨설팅 신청 유도 |

## 3. Red/Green 시스템 통합 디자인 규칙 (Visual Integration Rules)

Loss Gauge의 최종 결과(Red/Green)는 단순한 색상 표시를 넘어, **스토리텔링의 핵심**이 되어야 합니다.

1.  **Red 상태 진입 시:**
    *   배경색은 `#D32F2F`로 즉시 전환되어야 하며, 텍스트(Loss Amount)는 흰색으로 명확하게 대비되어야 합니다.
    *   시스템은 **"생존 모드 (Survival Mode)"**라는 스토리텔링 문구를 함께 노출하여 공포를 구체적인 행동 지침으로 전환합니다.

2.  **Green 상태 도달 시:**
    *   배경색은 `#3CB371`로 전환되며, 텍스트는 흰색 또는 밝은 회색으로 명확하게 표시됩니다.
    *   시스템은 **"성장 로드맵 (Growth Roadmap)"**이라는 문구를 노출하여 달성한 성과와 다음 단계의 기회를 강조합니다.

## 4. 최종 개발팀 전달 사항 (Action Items for Dev Team)

1.  **State Transition:** 모든 Loss Gauge 값 변화 시, 색상 전환 및 애니메이션은 **`200ms`**를 넘지 않도록 부드럽게 처리해야 합니다.
2.  **Feedback Clarity:** 경고(Warning Zone) 단계에서는 단순히 색상만 바꾸는 것이 아니라, 해당 영역에 맞는 구체적인 행동 제안 텍스트가 동적으로 표시되어야 합니다. (예: "다음 단계로 가기 위해 [A 전략]을 검토하세요.")
3.  **Component Structure:** Loss Gauge 컴포넌트는 **`Value Proof Flow`**의 각 단계(Pain Point -> Strategy -> Result)와 시각적으로 연결되어, 사용자가 '어디에 있는지' 뿐만 아니라 '어떻게 가야 하는지'를 동시에 인지하도록 설계해야 합니다.
