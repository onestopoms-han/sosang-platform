# 📄 BDS 플랫폼 캐릭터 통합 디자인 핸드오버 패키지 (v1.0)

## 1. 디자인 목표 및 시스템 요약

**목표:** '옆집선배' 캐릭터를 단순한 장식이 아닌, 데이터 기반 컨설팅을 안내하는 **신뢰성 있는 멘토**로서 기능하도록 시각적 일관성을 확보하고, 사용자 여정(온보딩, 분석)에 따라 적절히 배치하여 신뢰도를 극대화한다.

**핵심 디자인 시스템 재확인:**
*   **Primary Color (신뢰):** Deep Blue (`#1A237E`) - 플랫폼의 안정성과 전문성을 상징.
*   **Accent Color (성장/안내):** Growth Green (`#4CAF50`) - 긍정적인 변화와 성장을 유도하는 안내선 역할.
*   **Typography:** 산세리프 계열 (예: Pretendard 또는 Noto Sans) 기반으로 미니멀한 라인 아트 디자인 준수.

## 2. 캐릭터 배치 전략 및 시각적 가이드라인

캐릭터는 플랫폼의 신뢰도를 높이는 **Trust Widget**의 확장된 형태로 기능하며, 모든 곳에서 일관된 인상을 주어야 합니다.

### A. 온보딩(Onboarding) 섹션 배치 가이드

| 위치 | 목적 | 시각적 구현 상세 | 상호작용 (Interaction) |
| :--- | :--- | :--- | :--- |
| **첫 화면 중앙 (Hero Area)** | 환영 및 신뢰 구축 | 캐릭터가 중심에 배치되며, 주변은 Deep Blue 배경으로 안정감을 제공. 라인 아트 스타일 강조. | 마우스 오버 시 미세한 움직임(Idle → Talking)을 통해 환영 메시지를 전달. |
| **온보딩 단계별 (Step-by-Step)** | 과정 안내 및 동기 부여 | 각 단계의 성공 지표 옆에 캐릭터가 '다음 단계'를 가리키는 형태로 배치. Growth Green으로 강조하여 긍정적 흐름 유도. | 특정 정보 입력 완료 시, 캐릭터가 다음 단계로 자연스럽게 이동하는 애니메이션 구현. |
| **신뢰 위젯 (Trust Widget)** | 신뢰도 강화 | Trust Widget 내부에 마스코트의 작은 아이콘 형태로 배치. 데이터 분석에 대한 멘토링을 암시. | 클릭 시 캐릭터의 대화창(Talking State)이 나타나 핵심 통계 요약 제공. |

### B. 데이터 분석 섹션 배치 가이드 (Dashboard/Analytics)

| 위치 | 목적 | 시각적 구현 상세 | 상호작용 (Interaction) |
| :--- | :--- | :--- | :--- |
| **대시보드 상단 (Header)** | 컨설팅 접근성 제공 | 미니멀한 라인 아트 형태로, 항상 상단에 고정(Sticky Position). 데이터 분석 결과 옆에 배치하여 '해석'의 의미 부여. | 캐릭터가 특정 지표(KPI) 위에 떠서 해당 수치에 대한 간략한 코멘트(Tooltip)를 제공. |
| **심층 분석 모듈 (Deep Dive Module)** | 복잡한 데이터 해설 | 복잡한 그래프나 통계 데이터 옆에 캐릭터의 대화 상자를 배치하여, "이 데이터는 의미하는 바가..."와 같은 컨설팅 메시지를 시각적으로 삽입. | 사용자가 특정 차트를 클릭했을 때, 관련 분석 내용에 대한 캐릭터의 설명(Listening/Talking State)을 팝업 형태로 표시. |

## 3. 개발팀 핸드오버 패키지 (최종 명세)

### 📦 컴포넌트 요구사항: `NextdoorSeniorCharacter`

**기술적 목표:** CoDary가 요구한 상태 머신(Idle → Talking → Listening → Error) 로직을 완벽히 구현하며, 디자인 가이드라인을 따르는 시각적 표현이 필수.

1.  **스타일링 (Visual Spec):**
    *   캐릭터는 **Line Art 스타일**로만 표현한다. 채색은 최소화하고, Deep Blue 색상으로 외곽선(Outline) 처리하며 Growth Green을 강조선으로 사용한다.
    *   모든 상태(`idle`, `talking`, `listening`, `error`)에 따라 캐릭터의 포즈와 표정(라인 아트 내의 단순한 형태 변화)이 명확하게 정의되어야 한다. (이는 `Session/2026-06-17T12-56/designer.md` 기반으로 구체화 필요).
    *   **레이아웃 좌표:** 모든 배치 위치는 반응형 디자인을 고려하여, 페이지의 시각적 계층 구조(Visual Hierarchy)를 해치지 않는 영역에 위치해야 한다.

2.  **상태별 UI/UX 매핑:**

| 상태 (State) | 시각적 표현 | 기능적 의미 | 관련 API 연동 |
| :--- | :--- | :--- | :--- |
| **Idle** | 정적인 라인 아트 포즈 | 대기 및 준비 상태. 플랫폼의 안정성 반영. | `onSpeak` / `onListen` 비활성화 |
| **Talking** | 입 모양과 함께 텍스트 박스 활성화 | 정보 전달 시작. 사용자에게 집중 요구. | 외부 텍스트 입력 대기 |
| **Listening** | 귀를 기울이는 듯한 포즈, 부드러운 애니메이션 | 데이터 수집 및 이해 중. 능동적인 경청 상태. | `onListen` 이벤트 활성화 |
| **Error** | 경고 아이콘 (Growth Green 사용)과 함께 멈춤 | 데이터 오류 또는 시스템 문제 발생. 즉각적인 주의 환기. | API 오류 코드 반환 시 트리거 |
