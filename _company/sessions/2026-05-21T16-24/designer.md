## 성장 로드맵 및 ROI 섹션 개발 통합 사양서 (Design & Integration Specification)

### 1. 최종 UI/UX 흐름 명세 (Journey Map & Wireframe Specification)

**목표:** 소상공인의 성장 여정을 시각화하고, 데이터 기반의 인과관계를 명확히 제시하여 행동을 유도한다.

| 단계 | 사용자 목표 | 시스템 액션 (UI 표시) | 핵심 데이터 요구사항 | 상호작용 규칙 (Interaction Rule) |
| :--- | :--- | :--- | :--- | :--- |
| **1. 현재 상태 진단** | 내 성장의 현황 파악 | 전체 로드맵 진척도(%) 및 현재 위치 표시. 핵심 KPI 요약 카드 노출. | 현재까지의 데이터 누적, 목표 대비 달성률 (%), 주요 병목 지점. | 클릭 시 해당 단계 상세 분석 페이지로 이동 (Drill-down). |
| **2. 성장 로드맵 선택** | 다음 성장 경로 결정 | 타임라인 기반의 단계별 진행 상황 표시. 각 단계(예: 마케팅 강화, 시스템 구축)에 대한 명확한 목표 제시. | 각 단계별 목표 설정 값 및 예상 소요 시간/비용 (ROI 연동). | 드래그 앤 드롭 또는 선택 방식으로 다음 단계 진입을 유도. |
| **3. ROI 시뮬레이션** | 투자 대비 효과 예측 | 선택된 성장 경로에 따른 잠재적 ROI 변화를 그래프로 실시간 시각화. | 입력된 가설 데이터(투자 금액, 예상 결과) 기반의 예측 모델 결과. | 사용자가 변수를 변경할 때마다 즉시 (Real-time) 그래프 업데이트. |
| **4. 실행 계획 확인** | 구체적인 다음 행동 정의 | 선택된 경로에 필요한 세부 액션 아이템(Task List)과 연계된 실행 단계 제시. | 각 액션 항목별 필수 데이터 입력 필드 및 연동 API 명세. | '다음 단계 진행' 버튼 클릭 시, 해당 액션의 입력 폼으로 자동 이동. |

### 2. 데이터 시각화 및 KPI 정의 (Visualization & KPI Definition)

**핵심 원칙:** 신뢰(Trust), 성장(Growth), 명확성(Clarity)을 전달하는 Teal 색상 기반 시스템 유지.

| 지표 그룹 | 핵심 KPI 항목 | 시각화 방식 | 디자인 가이드라인 | 데이터 소스 연동 |
| :--- | :--- | :--- | :--- | :--- |
| **성장 추이** | 누적 성장률, 월별/분기별 변화 | 선 그래프 (Line Chart) | Teal 색상 사용. 목표 달성 시 녹색 하이라이트. | 백엔드 ROI 및 성장 데이터 API |
| **자원 효율** | 투자 대비 수익률 (ROI) | 도넛 차트 또는 게이지 미터 | ROI 값에 따라 색상 변화 (녹색: 150% 이상, 노란색: 100-150%, 빨간색: 100% 이하). | `calculate-roi` API 결과 |
| **병목 분석** | 핵심 병목 지점 (Bottlenecks) | 트리맵 또는 단계별 막대 차트 | 가장 낮은 성과를 보이는 구간을 명확히 구분하여 강조. | 사용자 입력 데이터 및 시스템 제약 조건 |

### 3. 통합 테스트 시나리오 기반 데이터 연동 지침 (Integration Guidelines)

개발팀이 `POST /api/v1/analytics/calculate-roi` 호출 및 데이터 수신에 필요한 구체적인 규칙입니다.

**A. 입력 데이터 스키마 (Input Data Schema for API Call)**

| 필드명 | 타입 | 필수 여부 | 설명 | 제약 조건 (Validation) |
| :--- | :--- | :--- | :--- | :--- |
| `roadmap_data` | Array of Objects | 필수 | 성장 로드맵 단계별 데이터 배열. | 각 객체는 `stage`, `investment`, `expected_outcome` 포함해야 함. |
| `baseline_metric` | Number | 필수 | 초기 기준 지표 (예: 초기 매출액). | 양수( > 0)여야 함. |
| `time_horizon` | String | 필수 | 분석하고자 하는 기간 (예: '6개월', '12개월'). | 사전 정의된 기간 리스트에서 선택해야 함. |

**B. 백엔드 응답 및 프론트엔드 매핑 규칙 (Response & Mapping)**

*   **성공 응답 (HTTP 200 OK):**
    *   응답 필드는 반드시 다음 구조를 따라야 하며, 모든 값은 **소수점 둘째 자리**까지 반올림되어야 합니다.
    ```json
    {
      "status": "success",
      "calculated_roi": 185.45, // 실제 ROI 값
      "growth_trend": {
        "overall_change": "+25%", // 전체 성장률
        "stage_breakdown": [
          {"stage": "Marketing", "roi": 190.1},
          {"stage": "System_Build", "roi": 178.9}
        ]
      },
      "error_notes": null // 에러가 없을 경우 null 반환
    }
    ```

*   **에러 응답 (HTTP 4xx/5xx):**
    *   만약 입력 데이터의 유효성 검사 실패(예: `baseline_metric`이 음수) 시, `error_notes` 필드에 구체적인 오류 메시지를 반환해야 합니다. 프론트엔드는 이 메시지를 받아 사용자에게 명확하게 피드백해야 합니다.

**C. 데이터 무결성 및 UX 피드백 (Data Integrity & UX Feedback)**

1.  **Negative Testing:** 개발팀은 최소 5가지의 **'잘못된 입력 시나리오'** (예: 비현실적인 ROI 요구, 잘못된 시간 범위 입력)에 대해 API 호출을 테스트하고, 시스템이 예상대로 오류 메시지를 반환하는지 확인해야 합니다.
2.  **Visual Feedback:** 백엔드에서 특정 KPI가 계산 불가능한 상태로 반환될 경우(예: 데이터 부족), 프론트엔드는 로딩 상태 대신 **"데이터 분석에 필요한 정보가 부족합니다. 다음 단계를 확인해주세요."**와 같은 구체적인 UX 메시지를 표시해야 합니다.

### 4. Mockup 최종 파일 지침 (Figma/Sketch Instruction)

개발팀이 참고할 수 있도록, Figma 내 해당 섹션의 레이아웃과 컴포넌트 명명 규칙을 확정합니다.

*   **레이아웃:** 전체 섹션은 좌측에 **'Progress Timeline'**을 배치하고 우측에 **'ROI Visualization Panel'**을 배치하는 2단 구조를 유지한다.
*   **색상 사용:** Teal (Primary), Gray-50 (Background), Accent Green/Red (Status Indicator)만을 사용하여 정보의 위계를 명확히 한다.
*   **컴포넌트 명명 예시:** 모든 차트는 `ROI_LineChart` 또는 `Growth_DonutMeter`와 같이 기능 기반으로 컴포넌트를 명명하여 재사용성을 높인다.
