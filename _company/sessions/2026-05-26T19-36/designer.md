**작성자:** BDS Designer Lead
**목표:** 모든 기술적 예외 상황(Failure State)에 대해, 시스템의 신뢰성 지표(PRSR/ERT 등)를 기반으로 정의된 **최소 허용 UX 기준(Minimum Viable Trust)**을 충족하는지 검증한다. 이 매트릭스는 최종 QA 및 승인 보고서의 핵심 근거 자료로 활용된다.

---
## 🔍 Verification Scope: 시스템 신뢰 컴포넌트 (System Trust Component)
**적용 범위:** 결제, 데이터 전송, 서버 응답 지연 등 모든 비동기/예외 상황 발생 시.
**핵심 원칙:** **"기술 실패 → 사용자의 통제감 상실 → 디자인으로 인한 회복(Recovery)"**의 흐름을 증명한다.

### 📋 체크리스트: 기술-경험 연계 검증 항목 (Tech-to-UX Linkage Checklist)

| 구분 | 시스템 실패 지표 (Metric Failure) | 발생 상황 및 예외 Flow | UX 목표/가설 (User Goal) | 필수 디자인 요구사항 (Design Specification) | 테스트 결과 기대치 (Pass Criteria) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **I. 결제 흐름 실패** | PRSR (Payment Retry Success Rate) 저하 / ERT (Error Tolerance Time) 초과 | 1. 네트워크 불안정으로 인한 결제 API 호출 실패 <br> 2. 서버 과부하로 인한 PG 연동 오류 | 사용자가 '내가 원인 제공자가 아니다'라고 인식하고, **쉽고 빠른 재시도 경로**를 찾는다. (통제감 회복) | - 즉각적인 피드백: "잠시만요" 메시지 및 애니메이션 처리 <br> - 상세 안내: 실패 원인을 기술 용어 없이 간결하게 설명 (< 10자). <br> - Actionable Button: **'자동 재시도 (권장)'** 버튼 활성화. | ① 오류 발생 시 지연 시간(Latency)이 UX 허용 시간(3초)을 넘기지 않아야 한다. ② 실패 화면에서 '재도전' 외 다른 선택지가 없어야 한다. |
| **II. 데이터 동기화 실패** | Data Sync Interval Timeout (데이터 동기화 간격 초과) | 1. 로컬 데이터와 서버 데이터의 불일치 발생 <br> 2. 네트워크 단절로 인한 정보 업데이트 실패 | 사용자가 현재 자신의 기기가 '신뢰할 수 있는 최신 정보'를 보여주고 있다고 확신하게 한다. (정확성 확보) | - 명시적 상태 표시: "데이터 동기화 중" 또는 "최종 확인 시간: MM/DD HH:MM". <br> - 경고 메시지: 단순히 오류가 아닌, **불일치의 원인**을 설명하고 '새로고침' 유도. | ① 데이터 불일치 발생 시, 사용자에게 최신 정보의 출처와 시간을 명시적으로 보여줘야 한다. ② 강제 새로고침(Pull-to-Refresh) 시 로딩 상태가 멈추지 않고 부드럽게 진행되어야 한다. |
| **III. 플랫폼 비활성/오류** | Session Timeout / Component Crash (세션 만료 또는 컴포넌트 충돌) | 1. 사용자가 장시간 페이지에 머무르거나, 브라우저가 강제 종료될 경우 <br> 2. 시스템 내부 로직 오류로 인한 UI 깨짐 발생 | 사용자가 플랫폼이 **'항상 자신을 지켜보고 있다'**는 느낌(Presence)을 받고, 재로그인 과정의 불편함을 최소화한다. (지속적 연결감 확보) | - Graceful Degradation: 핵심 기능 위주로만 화면 구성 <br> - 복구 메커니즘: '최근 작업 항목 미리보기' 제공 및 **자동 세션 유지** 시도 로직 구현. | ① 시스템 충돌 발생 시, 에러 코드가 노출되어서는 안 되며, 대신 "시스템 점검 중입니다"와 같은 친절한 문구로 대체되어야 한다. ② 로그아웃 직전의 작업 내용은 반드시 임시 저장(Draft) 상태로 남아있어야 한다. |

### 📑 핵심 증거물 요약 및 추가 검토 항목 (Final QA Checklist)

1.  **[필수] 에러 메시지 구조 표준화:** 모든 오류 메시지는 다음 3요소를 포함해야 합니다.
    *   `① 무엇이 문제인가?` (예: 결제에 실패했습니다.) - **쉬운 언어**
    *   `② 왜 문제가 발생했는가?` (예: 카드사 전산망 점검 중일 수 있습니다.) - **원인 설명 (불안 해소)**
    *   `③ 어떻게 해결해야 하는가?` (예: 5분 후 재시도해 주세요.) - **명확한 액션 플랜**

2.  **[필수] PRSR/ERT 모니터링 대시보드 연동:** 최종 사용자에게는 보이지 않지만, 개발자/관리자 페이지에는 위에서 언급된 모든 실패 지표(PRSR, ERT 등)가 실시간으로 누적되고 시각화되는 **'안정성 대시보드'**를 반드시 구축하여 증거 자료로 확보해야 합니다.
