**목표:** 모든 시각 요소가 '기술적 실패 가능성이 낮다'는 확신과 안도감을 주는 방향으로 설계되었는지 최종 검증한다. (Success Rate 95%, Retry Rate 80% 반영)

| :---: | :--- | :--- | :---: | :--- |
| **1.1** | **톤앤매너 (Tone)** | 불안감 유발 금지. 모든 문구는 '불안 해소' 및 '확신 부여'에 초점을 맞춘다. (예: "문제가 발생할 수 있습니다" $\rightarrow$ "**걱정 마세요.** 저희가 자동으로 처리합니다.") | | 긍정적이고 능동적인 단어 선택 필수. |
| **1.2** | **컬러 활용 (Color)** | 실패/경고 시 Red 계열을 최소화하고, 시스템 안정성 메시지 전달 시에는 Deep Blue(#004D66)를 메인으로 사용한다. 성장은 Growth Green(#3CB371)을 유지하되, '안정'은 Blue로 지배한다. | | **에러 상태의 Primary Color는 밝은 주황색(Warning Orange)**을 사용하고, Deep Blue 배경 위에 배치하여 경고와 신뢰를 분리해야 함. |
| **1.3** | **핵심 메시지 (Message)** | 기술적 안정성 지표(95%, 80%)가 단순 숫자가 아니라 '금전적 이득'으로 연결됨을 명확히 한다. (예: "안정성 보험료 = 투자 대비 회수율 증가분") | | 대시보드 상단에 **‘안정성 프리미엄’** 섹션을 별도로 할애하여 지표를 시각화해야 함. |
| **1.4** | **레이아웃 (Layout)** | 정보를 한 번에 너무 많이 제공하지 않는다. 가장 먼저 눈에 들어오는 것이 '현재 상태의 안정적 작동 여부'여야 한다. | | 필수 데이터(Success Rate)는 항상 화면 최상단, 중앙 정렬 배치. |

### 🔵 Part 2. 대시보드 (Dashboard) 검토 체크리스트: [안정성 증명]

| # | UI/UX 요소 | '안정성' 메시지 구현 방식 (How to Show Stability) | Y/N | 비고 (필수 수정 사항) |
| :---: | :--- | :--- | :---: | :--- |
| **2.1** | **대시보드 헤더** | 플랫폼의 핵심 가치(신뢰)를 담은 강력한 슬로건 배치. (예: "데이터 기반 성장, 흔들림 없는 성공을 약속합니다.") | | 타이포그래피에 깊이감 있는 세리프체를 활용하여 신뢰감을 높인다. |
| **2.2** | **지표 섹션 (KPI)** | Success Rate 95%와 Retry Logic 80%를 단순히 그래프가 아닌, **'보장된 성공률'**이라는 문구로 감싸서 보여줘야 한다. | | 지표 아래에 "이 수치는 시스템의 지속적인 모니터링을 통해 보장됩니다."라는 *미세한 신뢰 문구*를 반드시 삽입한다. |
| **2.3** | **ROI 시뮬레이션** | 결과값이 '확률적'이지 않고, 계산된 '수익 예측치'처럼 보이게 한다. (금전적 안정성 강조) | | Growth Green을 사용하여 성장 기대치를 보여주되, Deep Blue 배경과 대비시켜 신뢰를 해치지 않도록 한다. |
| **2.4** | **CTA 영역** | 행동 유도(신청/구매) 버튼에 '안정적인 다음 단계'라는 느낌을 부여한다. (예: "성장 로드맵 확정하기" 대신 "**95% 성공률로 확실하게 시작하기**") | | CTA 버튼 주변에 보호막 같은 시각적 요소(Shield Icon 등)를 추가하여 방어적 안정감을 준다. |

### 🔴 Part 3. 결제 실패/에러 화면 (Failure Screen) 검토 체크리스트: [안정성 복구]

| # | UI/UX 요소 | '안정성' 메시지 구현 방식 (How to Restore Stability) | Y/N | 비고 (필수 수정 사항) |
| :---: | :--- | :--- | :---: | :--- |
| **3.1** | **최초 노출 메시지** | 실패 원인을 사용자 탓으로 돌리지 않는다. 시스템의 *잠재적 문제*를 언급하고, 이를 해결하는 주체가 '시스템 자체'임을 명확히 한다. (예: "일시적인 네트워크 오류" 대신 "**데이터 처리 과정에서 일시적인 변동이 감지되었습니다.**") | | 패닉 상태의 사용자를 위한 **진정시키는 톤**을 최우선으로 한다. |
| **3.2** | **재도전 메커니즘 강조** | Retry Logic (80%)에 대한 언급을 가장 먼저, 그리고 크게 배치한다. 실패를 두려워하지 않도록 시스템이 '자동적으로 커버'하고 있다는 느낌을 주어야 한다. | | "걱정 마세요. 저희 80% 자동 재시도 로직이 작동합니다." 문구를 **가장 눈에 잘 띄는 곳(H1/H2 레벨)**에 배치한다. |
| **3.3** | **대안 제시 (Fallback)** | 단순히 '다시 시도' 버튼만 주지 않는다. *왜 실패했는지*의 원인 분석 및 다음 단계 가이드를 제공하여 통제감을 되찾아준다. | | "다음 단계를 위해 [원인 분석]을 먼저 확인해보세요."와 같이 **문제 해결 과정**을 안내해야 한다. |
| **3.4** | **시각적 요소** | 에러 코드가 나열되는 대신, 부드럽게 깜빡이는 애니메이션이나, 시스템이 '진단 중'이라는 느낌의 미니 UI를 삽입하여 기계적인 안정감을 시각적으로 제공한다. | | 텍스트 중심의 딱딱한 실패 화면은 피해야 한다. **움직임(Motion)**을 활용해 생동감 있는 신뢰를 구축한다. |
