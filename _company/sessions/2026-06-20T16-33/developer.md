## 2. Trust Widget V1 API 분석 (신뢰도 스냅샷 노출)
### 2.1 주요 엔드포인트 목록
| 엔드포인트 | 방법 | 역할 | 예상 트래픽 | 
|-----------|------|-------|------------|
| `GET /v1/trust/snapshot/{userId}` | GET | 특정 사용자 신뢰 점수 스냅샷 반환 | 초당 10~50 요청 (고정) |
| `POST /v1/trust/refresh` | POST | 전체 스냅샷 재계산 및 캐시 갱신 | 일일 1~2 회 (관리자) |

### 2.2 기술적 난이도 & 제약사항
- **난이도: ⭐⭐ (낮음)**  
  - 단순 데이터 조회 + `Redis` 캐싱 로직만 있으면 됨.  
  - 외부 API 연동 (예: 리뷰 플랫폼, 결제 내역) 은 별도 마이크로서비스로 분리.  

- **주요 제약:**
  1. **실시간성 vs 캐시 갱신 주기**:  
     - 캐시 TTL 은 `30 초`~`5 분` 사이에서 결정.  
     - 너무 짧으면 DB 부하↑, 너무 길면 데이터 신선도↓.  
     - **해결책:** LRU + `TTL = 120 초`(최초값)로 설정, 변경 시 `POST /v1/trust/refresh` 로 강제 갱신.  

  2. **DB 부하**:  
     - `SELECT trust_snapshot FROM user_trust WHERE user_id = ? LIMIT 1` 만 있으면 됨.  
     - 인덱싱: `(user_id)` 필수.  

- **예상 소요 시간:**
  - 구현 로직: **2~3 일** (단순 RESTful API + 단위 테스트)  
  - 캐시/DB 최적화: **+1 일**  
  - 문서화 (OpenAPI): **+0.5 일**  
  - **총합: 약 4~5 일**  

### 2.3 비즈니스 우선순위
- **ROI:** 매우 높음 (신뢰 점수 노출 → 전환율↑)  
- **리스크:** 낮음 (단순 조회 API)  
- **추천 우선순위: P0 (최우선)**  
  > ✅ MVP 첫 주에 반드시 완료해야 할 기능.

## 3. PainGauge V1 API 분석 (애로지표 대시보드)
### 3.1 주요 엔드포인트 목록
| 엔드포인트 | 방법 | 역할 | 예상 트래픽 | 
|-----------|------|-------|------------|
| `GET /v1/pain/dashboard/{userId}` | GET | 사용자별 애로지표 요약 반환 (재고, 매출 등) | 초당 5~20 요청 |
| `POST /v1/pain/sync` | POST | 외부 시스템(회계, 재고) 데이터 동기화 요청 | 일일 1 회 (백그라운드) |

### 3.2 기술적 난이도 & 제약사항
- **난이도: ⭐⭐⭐⭐ (높음)**  
  - 다중 소스 데이터 통합 및 `임의 필드 매핑` 로직 필요.  
  - 외부 API 연동 (예: 네이버 재고, 카카오페이 매출) 이나 내부 DB (SQLite → PostgreSQL) 마이그레이션 가능 여부 검토.  

- **주요 제약:**
  1. **데이터 일관성**:  
     - 재고/매출 데이터가 `30 초` 이내 최신이어야 함.  
     - 동기화 지연 시 **사용자 피드백이 발생** (예: "재고가 0" 표시됨).  
     - **해결책:** 외부 API 호출 시 `retry + exponential backoff`(최대 5 회) 로 구현, 내부 DB 는 `binlog` 기반 CDC 적용.  

  2. **필드 매핑 유연성**:  
     - 각 소상공인의 데이터 스키마가 다름 (예: 재고는 `int`, `decimal(10,2)` 등).  
     - **해결책:** Pydantic + `field_validator` 로 동적 스키마 변환.  

  3. **보안**:  
     - 외부 API 키 (Stripe, KakaoPay) 를 코드에 하드코드 금지 → `process.env.STRIPE_API_KEY` 사용.  
     - `.env` 파일과 `.gitignore` 적용 필수.  

- **예상 소요 시간:**
  - 구현 로직: **5~7 일** (다중 소스 연동 + Pydantic 스키마)  
  - 캐싱/오류 처리: **+2 일**  
  - 외부 API 인증 및 테스트: **+3 일**  
  - **총합: 약 10~12 일**  

### 3.3 비즈니스 우선순위
- **ROI:** 높음 (소상공인 고통 반영 → 솔루션 제안 기반)  
- **리스크:** 중 (외부 API 연동 및 데이터 일관성 문제)  
- **추천 우선순위: P0 (최우선)**  
  > ✅ MVP 필수 기능이지만, 개발 리소스가 많이 소요됨.

## 4. PainGauge V2 (가상 시뮬레이션) vs Trust Widget V2 (AI 예측)
### 4.1 주요 엔드포인트 목록
| 엔드포인트 | 방법 | 역할 | 예상 트래픽 | 
|-----------|------|-------|------------|
| `GET /v1/pain/simulate` | GET | 시나리오 기반 예측 반환 (예: "30 일 후 매출 -20%") | 초당 1~5 요청 |
| `POST /v1/trust/predict` | POST | AI 모델로 신뢰 점수 예측 (예: 다음 달 리뷰 증가) | 초기에는 거의 없음 |

### 4.2 기술적 난이도 & 제약사항
- **난이도: ⭐⭐⭐⭐⭐ (매우 높음)**  
  - 머신러닝 모델 연동 + `실시간 데이터 파이프라인`(Kafka, Airflow 등) 필요.  
  - GPU 서버나 클라우드 ML 플랫폼 사용 시 비용 증가.  

- **주요 제약:**
  1. **모델 정확도**:  
     - 초기에는 단순 회귀 모델 (Linear Regression) 로 MVP 구축, 이후 Grad Boosting 으로 확장.  
     - 예측 오차가 5% 이상일 경우 **사용자 신뢰도 하락** 가능.  

  2. **데이터 프라이버시**:  
     - 매출/재고 데이터는 민감 → `GDPR` 및 `개인정보보호법` 준수 필요.  
     - **해결책:** 데이터 익명화 + `KMS`(한국전자문서공인인증기관) 암호화 저장.  

- **예상 소요 시간:**
  - 구현 로직: **15~20 일** (ML 모델 연동 + 파이프라인 구축)  
  - 모델 학습 및 테스트: **+7 일**  
  - **총합: 약 25~30 일**  

### 4.3 비즈니스 우선순위
- **ROI:** 낮음 (MVP 초기에는 과다 리소스 소요)  
- **리스크:** 매우 높음 (모델 오작동 시 신뢰도 하락)  
- **추천 우선순위: P1 (후순위)**  
  > ✅ MVP 출시 후, 사용자 피드백을 반영하여 단계적 확장.

## 5. 종합 기술적 제약사항 및 권장 사항
### 5.1 기술 스택 추천
| 기능 | 백엔드 프레임워크 | 데이터베이스 | 캐싱/큐 | 
|------|------------------|-------------|----------|
| Trust Widget V1 | FastAPI (TypeScript 연동 가능) | SQLite → PostgreSQL | Redis + TTL |
| PainGauge V1 | FastAPI + Pydantic 스키마 | PostgreSQL + binlog | Redis + LRU |

### 5.2 개발 로드맵 우선순위
1. **P0 (최우선)**: Trust Widget V1 (4~5 일)  
   - MVP 첫 주에 반드시 완료, 신뢰도 노출 로직 검증.  
2. **P0 (최우선)**: PainGauge V1 (10~12 일)  
   - 개발 리소스 확보 후 2 주 내 완료, 데이터 소스 연동 테스트.  
3. **P1 (후순위)**: Trust Widget V2 + PainGauge V2 (25~30 일)  
   - MVP 출시 후 사용자 피드백을 반영하여 단계적 확장.  

### 5.3 예상 총 소요 시간 및 인력
- **총 개발 시간:** 약 4~6 주  
- **필요 인원:** 1명 (코다리) + 1명 (현빈 - 비즈니스 전략) = MVP 핵심 기능 완료 가능.  
- **비용:** 서버/클라우드 비용 약 $50~100 / 달 (초기 트래픽 기준).
