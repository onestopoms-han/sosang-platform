## 1. MVP 기능 우선순위 (Must-have vs Should-have)

개발 자원을 효율적으로 배분하기 위해, **Trust Widget** 과 **PainGauge** 의 기능을 MVP 로 포함할지 여부를 판단합니다. 단순한 UI 를 넘어 '데이터 기반 의사결정 도구'로서의 가치를 검증하는 것이 목표입니다.

### 1.1. 필수 기능 (Must-have - V1.0)
이 기능들은 플랫폼의 핵심 가치인 **'신뢰도 시각화'**와 **'위기 예방'**을 구현하며, MVP 에서 반드시 포함해야 합니다.

| 기능 명세 | 비즈니스 목적 | 개발 우선순위 | 의존성 |
| :--- | :--- | :--- | :--- |
| **Trust Widget (State Display)**<br>- 현재 신뢰도 상태(Green/Amber/Red) 시각화<br>- 간단한 행동 유도 메시지 표시 | 초기 사용자에게 '내가 안전한가?'를 즉시 인지하게 함.<br>대시보드 전환율 및 이탈률 개선. | P0<br>(가장 높음) | TrustWidgetData 스키마 완성 |
| **PainGauge (Risk Indication)**<br>- 주요 위험 지표(재정, 공급망 등) 3 개 항목의 수치 표시<br>- 위험 수준별 색상 연동 | 소상공인의 가장 큰 불확실성(위험)을 한눈에 파악하게 함.<br>AI 기반 솔루션 도입 설득. | P0<br>(가장 높음) | PainGauge 스키마 완성 |
| **Action Prompt (Simple)**<br>- 상태별 고정된 행동 유도 문구 표시 (예: "안전합니다", "주의 필요") | 사용자의 다음 행동을 유도.<br>트래픽 소스 분석 시 '행동' 기반 세분화 가능. | P1<br>(중) | ActionPrompt 스키마 완성 |

### 1.2. 기대 기능 (Should-have - V1.5 ~ V2.0)
MVP 출시 후 데이터 기반으로 검증된 뒤, 다음 버전으로 추가해야 할 기능입니다. 기술적 실행 가능성은 높으나 비즈니스 가치 우선순위가 현재는 낮습니다.

| 기능 명세 | 비즈니스 목적 | 개발 우선순위 | 의존성 |
| :--- | :--- | :--- | :--- |
| **Trust Widget (Historical Trend)**<br>- 과거 7 일간 신뢰도 변화 그래프<br>- 추이 예측(선형 회귀 등) 제공 | 장기적인 사업 안정성 평가.<br>투자자 유치 및 대출 신청 시 참고 자료. | P2<br>(낮음) | Historical Data API 연결 |
| **PainGauge (Detail Drill-down)**<br>- 위험 지표 클릭 시 상세 원인 및 해결책 AI 추천<br>- 관련 정책/보조금 정보 연동 | 심층적인 문제 해결.<br>AI 기반 솔루션 판매 기회 창출. | P2<br>(낮음) | Large Language Model API 연동 |
| **Multi-User View (Admin)**<br>- 관리자 대시보드에서 다수 소상공인 데이터 집계 및 이상치 감지 | 플랫폼 운영 효율성 향상.<br>신용평가 모델 학습용 데이터 확보. | P3<br>(가장 낮음) | Database Schema 확장 |

### 1.3. 우선순위 결정 근거
- **P0 (Must-have)**: MVP 의 핵심 가치인 '데이터 기반 안전망'을 구현하지 않으면 플랫폼의 존재 의미가 퇴색합니다. 개발 리소스를 집중해야 합니다.
- **Should-have**: 기능은 중요하지만, 시장 반응(사용자 행동 데이터)이 먼저 필요합니다. V1.0 을 출시한 후, 실제로 어떤 행동을 유도하는지 검증한 뒤 개발 자원을 투입합니다.

## 2. 시장 검증(A/B 테스트) 시나리오

MVP 기능을 도입했을 때 '어떤 효과를 얻을 수 있을까?'를 예측하고 측정할 수 있는 구체적인 A/B 테스트 계획을 수립합니다.

### 2.1. 테스트 목표
- **주요 지표 (Primary Metric)**: 대시보드 페이지에서의 **행동 유도 클릭률(CTR, Call-to-Action Click-Through Rate)**. 신뢰도 상태별 행동 유도가 실제 행동으로 이어지는지 확인.
- **부수적 지표 (Secondary Metrics)**: **평균 체류 시간**, **반복 방문 횟수**, **'솔루션 탐색' 페이지로의 전환율**.

### 2.2. 테스트 대상 및 기간
- **대상**: 초기 베타 사용자 (소상공인 50~100 명).  
- **기간**: MVP 출시 후 4 주 (최대 2 회 테스트 사이클).

### 2.3. A/B 테스트 시나리오 설계

#### 시나리오 1: Trust Widget 의 행동 유도 효과 검증
| 변형 | 설명 | 예상 결과 |
| :--- | :--- | :--- |
| **A (Control)**<br>- 현재 신뢰도 상태만 표시.<br>- 별도 행동을 유도하지 않음. | 기준군. 사용자는 상태를 인지하더라도 다음 행동을 고민하게 됨. | CTR: ~5% |
| **B (Test Group 1)**<br>- Green 상태 + "안전합니다. 지금 바로 솔루션을 시작하세요."<br>- Amber/Red 상태 + "위험이 발견되었습니다. AI 컨설턴트와 상담하기" | 긍정적 행동 유도 시 전환율 증가.<br>부정적 상황에서도 '상담'이라는 구체적인 행동을 유도. | CTR: ~10% (예상) |

#### 시나리오 2: PainGauge 의 시각적 강조 효과 검증
| 변형 | 설명 | 예상 결과 |
| :--- | :--- | :--- |
| **A (Control)**<br>- 숫자와 상태만 표시.<br>- 색상 대비가 낮음. | 사용자는 상태를 인지하더라도 위험도를 정확히 판단하지 못함. | 체류 시간: 1 분 |
| **B (Test Group 2)**<br>- 위험도별 배경색 강조 + 아이콘 추가 (예: ⚠️)<br>- 주요 위험 요소만 하이라이트.<br>
