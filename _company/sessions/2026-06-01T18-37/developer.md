# E2E 테스트 시나리오: Payment Failure Screen (V1.0)

## 1. 개요
**목표**: 결제 실패 상황에서의 UI/UX 흐름, 에러 카드 컴포넌트 반응성, 재시도 로직을 종합적으로 검증합니다.  
**테스트 환경**: 모의 결제 게이트웨이 (`MockPaymentGateway`) 를 활용한 단위 테스트 및 통합 테스트 (E2E).

## 2. 시나리오 상세 정의
### 시나리오 A: 새 카드 입력 → 유효성 검사 실패
- **입력 데이터**: `CardNumber`: "0000111122223333", `CVV`: "123", `Expiry`: "12/25"
- **기대 동작**:  
  1. 제출 시 즉시 유효성 검사 실패 메시지 표시 (Luhn 알고리즘 미합치).  
  2. 에러 카드 컴포넌트: `status="error"` 상태 유지, `message="카드 번호가 유효하지 않습니다."`
- **검증 포인트**: 실시간 입력 유효성 검사 로직과 에러 메시지 타이밍.

### 시나리오 B: 기존 저장된 카드 선택 → 결제 실패 처리
- **입력 데이터**: 저장된 카드 목록 중 1 번째 카드 클릭.  
- **모의 서버 응답**: `{"success": false, "error": {"code": "DECLINED", "message": "이 거래는 거부되었습니다."}}`
- **기대 동작**:  
  1. 에러 카드 컴포넌트: `status="error"` → 빨간색 배경 및 "결제 실패" 아이콘 표시.  
  2. `Retry` 버튼 표시 (클릭 시 다시 시도).  
  3. 다른 카드 선택 시 기존 저장된 카드 목록이 다시 로드됨.
- **검증 포인트**: 에러 상태 관리, 컴포넌트 렌더링 조건부 로직, 저장된 카드 데이터 연동.

### 시나리오 C: 네트워크 오류 시뮬레이션
- **입력 데이터**: `CardNumber`: "4111 1111 1111 1111" (테스트용 승인 카드로서 유효).  
- **모의 서버 응답 딜레이**: 랜덤 3~5 초 지연 후 정상 응답.  
- **기대 동작**:  
  1. 로딩 스키톨 (Skeleton) 표시 → 성공 시 카드 정보 업데이트.  
  2. 타임아웃 (5 초 초과) 발생 시 `Retry` 버튼 강제 활성화 및 자동 재시도 알림 메시지 표시.
- **검증 포인트**: 로딩 상태 UI, 타임아웃 처리 로직, 사용자 알림 타이밍.

## 3. 모의 결제 게이트웨이 (`MockPaymentGateway`) 요구사항
테스트 환경 구축을 위해 다음 기능을 구현해야 합니다:
- `POST /api/payment/test`: 카드 정보 및 거래 ID 를 받아 다양한 응답 (성공/실패/타임아웃) 을 반환합니다.  
- **응답 데이터 구조**:  
  ```json
  {
    "success": true,
    "transaction_id": "txn_1234567890",
    "error": null
  }
  // 또는 실패 시
  {
    "success": false,
    "transaction_id": null,
    "error": {
      "code": "DECLINED",
      "message": "이 거래는 거부되었습니다."
    }
  }
  ```
- **테스트용 유효성 검사 규칙**:  
  - `CardNumber`: 16 자리 숫자 (Luhn 알고리즘).  
  - `CVV`: 3 자리 숫자.  
  - `Expiry`: MM/YY 형식, 오늘 이후 날짜.

이 시나리오와 요구사항을 바탕으로 모의 결제 서버 스크립트 (`mock_payment_gateway.py`) 를 작성하겠습니다.
