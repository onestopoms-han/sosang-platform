## 🎯 BDS 플랫폼 기술적 난제 (구현 격차) 현실화 로드맵 v1.0
> **작성일:** 2026. 06. 21
> **작성자:** Researcher Agent
> **용도:** 다음 주 모닝 브리핑 자료 / 개발-디자인 간 조정안

### 1. 현황 분석: 디자인 명세 vs 실제 구현 로직

#### 1.1 시각적 명세 (Designer - Figma)
- **Trust Widget**:
  - 상태별 색상 매핑 정의 (Critical=Red, Warning=Orange, Normal=Green).
  - 감정 점수 (0~100) 를 실시간으로 업데이트하는 애니메이션 효과.
  - 사용자 행동 유도 프롬프트 (예: "지금 도움이 필요한가요?") 가 시각적으로 강조됨.
- **Pain Gauge**:
  - 위험 지표 (High/Medium/Low) 를 그래프 형태로 표현.
  - 색상 및 아이콘 변경을 통해 상태 전환 시 직관적인 피드백 제공.

#### 1.2 실제 구현 로직 (Developer - 코다리)
- **API 명세서 (`track_emotional_interaction`)**:
  - `Critical`, `Warning`, `Normal` 상태를 텍스트로만 전송하는 단순한 JSON 구조.
  - 색상 매핑은 프론트엔드에서 수동 처리 예정 (Hard-coded 값 사용).
  - 애니메이션 효과는 CSS class toggling 로 구현, 데이터 기반 동적 업데이트 없음.
- **컴포넌트 코드 (`trust_widget.tsx`, `pain_gauge.tsx`)**:
  - 현재 상태가 `Critical`일 때만 Red 색상 적용 (조건문 로직).
  - 실시간 감정 점수 업데이트는 5초마다 polling 방식으로 구현, 효율성 저하.
  - 사용자 행동 유도 프롬프트는 정적 텍스트 렌더링, 동적 컨텍스트 반영 불가.

### 2. 격차 식별: 주요 문제점

| 구분 | 디자인 명세 요구사항 | 실제 구현 상태 | 격차 유형 | 심각도 | 영향력 |
| :-- | :--- | :--- | :--- | :--- | :--- |
| **색상 매핑** | 상태별 동적 색상 변경 (애니메이션 포함) | Hard-coded 조건문 (CSS class toggle) | 기능 누락 | 중 | UX 신뢰도 저하 |
| **데이터 연동** | 감정 점수 실시간 동기화 | Polling 방식 (5초 간격) | 비효율성 | 중 | 성능 문제, 사용자 이탈 위험 |
| **프롬프트** | 동적 컨텍스트 기반 유도 메시지 | 정적 텍스트 | 로직 부재 | 경 | 사용자 참여도 저하 |
| **위험 지표** | High/Medium/Low 상태의 시각화 (그래프) | 단순 텍스트 렌더링 | 기능 누락 | 중 | 직관성 부족 |

### 3. 현실적인 로드맵: 격차 해결 방안

#### Phase 1: 즉각 수정 (24~48 시간 내)
- **색상 매핑 개선**:
  - Hard-coded 값 대신 디자인 명세에서 정의된 컬러 팔레트를 변수로 관리 (`const COLORS = { critical: '#FF5733', ... }`).
  - CSS class toggle 로직을 유지하되, 상태 변경 시 부드러운 전환 (transition) 효과를 적용.
- **데이터 연동 최적화**:
  - Polling 방식 대신 WebSocket 또는 Server-Sent Events (SSE) 를 도입하여 실시간 데이터 스트리밍 구현.
  - 프론트엔드에서 무거운 애니메이션은 Canvas/WebGL 기반 컴포넌트로 분리하여 성능 저하 방지.

#### Phase 2: 기능 완성 (1 주일 내)
- **동적 컨텍스트 프롬프트**:
  - 사용자 행동 로그와 감정 데이터를 분석하여, 상황에 맞는 유도 메시지 생성 로직 구현 (`generatePrompt(context)` 함수).
  - 정적 텍스트 대신 AI 기반 자연어 처리 모델과 연동하여 실시간 응답 제공.
- **위험 지표 시각화**:
  - High/Medium/Low 상태에 따른 그래프 컴포넌트 추가 (Chart.js 또는 Recharts 사용).
  - 색상 및 아이콘을 디자인 명세와 일치하도록 적용.

#### Phase 3: 시스템 통합 (2 주 내)
- **API 명세서 개선**:
  - `track_emotional_interaction` API 응답 구조를 디자인 명세에 맞게 확장 (색상 코드, 애니메이션 duration, 프롬프트 텍스트 포함).
  - 백엔드에서 감정 분석 로직을 강화하여 더 정확한 위험 지표 산출.
- **디자인 시스템 연동**:
  - Figma 프로토타입의 컴포넌트 명세 (JSON Schema) 를 개발팀이 참조할 수 있도록 API 문서에 포함.
  - 자동화된 UI 테스트 스크립트를 작성하여 디자인 변경 시 코드 리팩토링 자동화.

### 4. 예상되는 난제 및 해결책

| 난제 | 가능성 | 해결책 |
| :-- | :--- | :--- |
| **WebSocket/SSE 서버 부하** | 중 | 백엔드에서 감정 분석 결과를 캐싱하고, 프론트엔드는 데이터 변경 시만 폴링 (polling interval) 방식으로 연동. |
| **정적 텍스트 기반 프롬프트의 한계** | 경 | AI 모델 학습을 위해 사용자 행동 데이터를 수집하고, 이를 기반으로 새로운 프롬프트 패턴을 생성하는 파이프라인 구축. |
| **디자인 명세 변경 시 코드 수정 비용** | 중 | Figma와 개발팀 간의 자동화된 워크플로우 (예: Figma-to-Code 플러그인) 도입하여 디자인 변경 시 자동으로 코드 업데이트 제안. |
