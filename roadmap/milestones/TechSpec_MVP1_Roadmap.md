# 🚀 MVP 1 개발 기획: Tech Spec 및 로드맵 초안

## 🎯 1. 비즈니스 목표 (Business Goal)
*   **핵심 가설 검증:** 소상공인들이 '검증된 경로'를 통해 공급자에게서 고품질/스토리텔링이 강화된 상품을 구매할 의향(Willingness to Pay)이 있는지 정량적으로 확인한다.
*   **최소 기능 정의 (MVP Scope):** 초기에는 복잡한 결제 시스템이나 대규모 사용자 온보딩 대신, **'검증된 공급자 리스트업 및 상담 예약/견적 요청'** 기능에 집중하여 데이터와 관계를 확보하는 것에 초점을 맞춘다.

## 🛠️ 2. 기술 요구사항 정의 (Technical Specification)
MVP는 세 가지 핵심 모듈로 구성됩니다. 모든 개발은 이 3가지 모듈을 중심으로 이루어져야 합니다.

### A. 공급자(Producer/Supplier) 관리 시스템 [Backend & Admin]
| 기능 ID | 상세 기능 명세 (What) | 요구사항 레벨 | 기술적 고려 사항 | 비고 (검증 포인트) |
| :---: | :--- | :---: | :--- | :--- |
| **P-01** | 공급자 정보 등록/수정 및 검증서 업로드 기능 | 필수 | 인증(JWT), 파일 저장소 연동, 백엔드 유효성 검사 로직. | '검증의 근거'가 시스템적으로 기록되어야 함 (Proof of Origin). |
| **P-02** | 판매 가능 상품 정보 등록 및 스토리 첨부 | 필수 | 카테고리 기반 메타데이터 관리(JSON/DB), 텍스트 길이 제한, 이미지 업로드. | '스토리텔링'의 정량화가 핵심. 단순 사진이 아닌 서사 구조 유도. |
| **P-03** | 상담 요청 및 견적 리드 수신 기능 | 필수 | 사용자 ID 매핑, 알림/메일 발송 API 연동 (Slack 등), 상태(Pending $\rightarrow$ Accepted). | B2B 거래의 초기 접점 확보. |

### B. 소상공인 사용자 인터페이스 [Frontend - Web/Mobile]
| 기능 ID | 상세 기능 명세 (What) | 요구사항 레벨 | 기술적 고려 사항 | 비고 (검증 포인트) |
| :---: | :--- | :---: | :--- | :--- |
| **C-01** | 카테고리 기반 상품 검색 및 필터링 | 필수 | 백엔드 API 연동, 실시간 검색 기능(Autocomplete), 최소 3개 이상의 필터 옵션 제공. | 사용자의 니즈 패턴 분석 (가장 많이 조합되는 필터 확인). |
| **C-02** | '스토리/검증' 기반 상품 상세 페이지 | 필수 | P-02 데이터 연동. 단순 가격 외에, 공급자 스토리, 인증 과정(흐름도), 예상 가치 등의 섹션을 배치. | 사용자가 **왜 이 제품을 믿어야 하는지**를 설득하는 UI 구조가 중요. |
| **C-03** | 상담/문의 요청 기능 (리드 생성) | 필수 | C-02 페이지 내에서 즉시 버튼으로 제공, P-03로 데이터 전송 및 리드 기록. | 구매 전환 직전 단계의 행동 유도. |

### C. 플랫폼 관리자 대시보드 [Backend & Admin]
| 기능 ID | 상세 기능 명세 (What) | 요구사항 레벨 | 기술적 고려 사항 | 비고 (검증 포인트) |
| :---: | :--- | :---: | :--- | :--- |
| **A-01** | 리드(Lead) 관리 대시보드 | 필수 | C-03으로 유입된 모든 문의를 시간순/상태별로 표시. 담당자 배정 및 처리 상태 변경 기능. | 플랫폼의 핵심 역할: 신뢰 기반 중개지 역할 수행 검증. |
| **A-02** | KPI 추적 대시보드 (핵심 지표) | 필수 | 누적 방문자 수, 검색 필터 이용률, 리드 발생 건수(월별), 공급자 등록 증가율을 시각화. | 현빈의 주력 목표. 데이터 기반 의사결정 근거 제공. |

## 📅 3. 초기 개발 로드맵 초안 (Development Roadmap Draft)

**전제:** 전담 개발팀(백엔드, 프론트엔드)이 투입된다는 가정 하에 작성되었습니다. 총 기간은 최소 **8주**를 목표로 합니다.

### 🟢 Phase 1: 기반 구축 및 핵심 흐름 정의 (Weeks 1-3)
*   **목표:** 백엔드의 데이터 구조 완성 및 핵심 사용자 흐름(Producer $\rightarrow$ Platform $\rightarrow$ Consumer)의 연결 검증.
*   **주요 작업:**
    1.  **DB Schema 설계 최종 확정:** P-01, P-02에 필요한 모든 필드 정의 (특히 `Story`와 `Certification Proof` 구조).
    2.  **Admin Module 구현:** A-01(리드 관리) 및 P-01(공급자 등록 API)의 최소 기능 버전 개발 완료.
    3.  **Frontend 와이어프레임 구축:** C-02 (상품 상세 페이지)의 핵심 스토리 섹션만 먼저 목업하여 구조 테스트 진행.

### 🟡 Phase 2: MVP 완성 및 통합 검증 (Weeks 4-6)
*   **목표:** 모든 모듈이 연결되어 '검증된 상품을 보고 문의를 남기는' 완결된 경험 제공.
*   **주요 작업:**
    1.  **Frontend 구현 완료:** C-01, C-02, C-03 기능 통합 및 UI/UX 디자인 적용 (Designer 산출물 활용).
    2.  **KPI 대시보드 연동:** A-02가 실시간으로 데이터를 받아와 시각화하는 테스트 환경 구축 (코다리 역할 증대).
    3.  **E2E 테스트:** 실제 가짜 사용자 3명이 처음부터 끝까지 사이트를 이용하며 버그 및 흐름 점검.

### 🔴 Phase 3: 파일럿 테스트 준비 및 배포 (Weeks 7-8)
*   **목표:** 실제 소수 공급자(5~10곳)와 사용자 그룹을 대상으로 하는 베타/파일럿 운영 환경 조성.
*   **주요 작업:**
    1.  **사용자 온보딩 개선:** 최초 접속자를 위한 가이드 및 스토리텔링 메시지 강화 (Writer/Instagram 산출물 활용).
    2.  **모니터링 시스템 구축:** 서비스 이용 로그(어디서 이탈하는가?)를 추적할 수 있는 GA 또는 유사 분석 툴 연동.
    3.  **최종 점검 및 배포 준비:** 보안 취약점 점검, 성능 최적화, 공식 출시 전 마지막 검토 완료.
