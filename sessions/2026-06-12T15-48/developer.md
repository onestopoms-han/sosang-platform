# 💻 코다리 — 백엔드 API 응답 지연/오류 시나리오 및 프론트엔드 상태 관리 명세 (V1.0)

## 1. 개요 및 목표
단순 진단 결과를 넘어서 사용자에게 감동적이고 설득력 있는 유료 전환 경험을 제공하기 위해, **백엔드 API 응답 지연/오류 시나리오를 구체화**하고 프론트엔드의 `isLoading` 상태 관리 조건을 정의해야 합니다. 이는 디자인팀의 '손실 최소화' 테마 에러 메시지 가이드라인과 완벽히 동기화되어야 합니다.

## 2. 백엔드 API 응답 지연/오류 시나리오 (Back-End Scenario)
백엔드 API 에서 발생할 수 있는 주요 응답 지연 및 오류 시나리오를 정의하고, 각 시나리오가 프론트엔드에 미치는 영향을 분석합니다.

### 2.1. 네트워크 지연 (Network Latency)
- **시나리오:** 백엔드 서버에서 DB 쿼리 실행 중 네트워크 지연이 발생하거나, 외부 API 호출 (예: third-party service) 이 느리게 응답하는 경우.
- **예상 소요 시간:** 200ms ~ 5s (평균 1.5s).
- **프론트엔드 영향:** `isLoading` 상태 유지, Skeleton Loader 애니메이션 진행, Fallback 로직 비활성화.

### 2.2. 백엔드 서버 과부하 (Server Overload)
- **시나리오:** 백엔드 서버 CPU/메모리 사용량이 높아 응답 속도가 느려지거나, 일부 요청은 타임아웃이 발생하는 경우.
- **예상 소요 시간:** 1s ~ 10s 이상 (불규칙).
- **프론트엔드 영향:** `isLoading` 상태 유지, Skeleton Loader 애니메이션 반복, Fallback 로직 부분적 활성화 (예: 기본값 표시).

### 2.3. 백엔드 API 오류 (API Error)
- **시나리오:** 백엔드 서버에서 500 Internal Server Error, 4xx Client Error 등 예상치 못한 에러가 발생하는 경우.
- **예상 소요 시간:** 즉시 또는 짧은 응답 (50ms ~ 2s).
- **프론트엔드 영향:** `isLoading` 상태 해제, `onError` 이벤트 발생, Fallback 로직 활성화 (예: 에러 메시지 표시).

### 2.4. 백엔드 API 타임아웃 (API Timeout)
- **시나리오:** 백엔드 서버 응답 시간이 설정된 시간 (예: 5s) 을 초과하여 타임아웃이 발생하는 경우.
- **예상 소요 시간:** 5s + (추가 대기).
- **프론트엔드 영향:** `isLoading` 상태 해제, `onTimeout` 이벤트 발생, Fallback 로직 활성화 (예: 기본값 또는 에러 메시지 표시).

## 3. 프론트엔드 `isLoading` 상태 관리 조건 (Front-End State Management)
백엔드 API 응답 지연/오류 시나리오에 대응하기 위해 프론트엔드의 `isLoading` 상태를 어떻게 관리할지 정의합니다.

### 3.1. 상태 전환 로직
- **초기:** `loading = true`, `error = false`, `data = null`.
- **성공 응답 (2xx):** `loading = false`, `error = false`, `data = response.data`.
- **클라이언트 오류 (4xx):** `loading = false`, `error = true`, `data = null` (또는 에러 정보).
- **서버 오류/타임아웃 (5xx 또는 timeout):** `loading = false`, `error = true`, `data = null` (또는 기본값).

### 3.2. Fallback 로직 활성화 조건
백엔드 API 응답 지연/오류 시나리오에 따라 프론트엔드의 Fallback 로직을 어떻게 활성화할지 정의합니다.

- **Skeleton Loader:** `loading === true`일 때 항상 애니메이션 진행, Fallback 로직 비활성화.
- **기본값 표시:** `loading === false && error === true && data === null`일 때 기본값 (예: "데이터 없음") 또는 에러 메시지 표시.
- **에러 메시지 표시:** `error === true`이고 백엔드 API 에서 에러 정보를 반환한 경우, 해당 에러 메시지를 표시.

### 3.3. 시각적 일관성 보장
백엔드 API 응답 지연/오류 시나리오가 프론트엔드의 시각적 일관성을 어떻게 보장할지 정의합니다.

- **Skeleton Loader 색상:** `loading === true`일 때 Skeleton Loader의 색상은 디자인 시스템의 `--color-skeleton`에 정의된 값과 일치해야 합니다.
- **에러 메시지 스타일:** `error === true`일 때 에러 메시지의 스타일은 디자인 시스템의 `--color-error`와 `--font-size-smaller`를 기반으로 해야 합니다.
- **기본값 표시 스타일:** 기본값을 표시할 때, 디자인 시스템의 `--color-neutral-muted`와 `--font-size-smaller`를 적용합니다.

## 4. 기술적 검증 및 리스크 (Technical Verification & Risks)
백엔드 API 응답 지연/오류 시나리오가 프론트엔드의 시각적 일관성을 어떻게 보장할지 기술적으로 검증하고, 잠재적인 리스크를 분석합니다.

### 4.1. 시각적 일관성 검증
- **Skeleton Loader 애니메이션:** Skeleton Loader의 애니메이션 타이밍과 색상 전환 로직이 디자인 시스템과 완벽히 일치하는지 확인.
- **에러 메시지 가이드라인:** 백엔드 API 에서 반환된 에러 메시지가 디자인팀의 '손실 최소화' 테마 에러 메시지 가이드라인을 준수하는지 검증.

### 4.2. 잠재적 리스크
- **네트워크 환경 변동성:** 네트워크 지연이 예상보다 길어질 경우, `isLoading` 상태가 오래 유지되어 사용자 경험이 저하될 수 있음.
- **백엔드 서버 과부하:** 백엔드 서버 과부하로 인해 API 응답 시간이 불규칙하게 증가할 경우, 프론트엔드의 Fallback 로직이 적절히 작동하지 않을 수 있음.
- **에러 메시지 처리:** 백엔드 API 에서 반환된 에러 메시지가 예상치 못한 형태로 전달될 경우, 프론트엔드가 이를正确处理하지 못해 사용자 경험이 저하될 수 있음.
