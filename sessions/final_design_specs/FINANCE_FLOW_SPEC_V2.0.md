# BDS 플랫폼 - 금융거래 흐름 최종 디자인 시스템 사양서 (V2.0)\n\n## 🎯 목적: 사용자 신뢰 유지 및 명확한 에러 핸들링 표준화\n\n본 문서는 코다리가 설계한 모든 예외 처리 로직(Fallback State)과 Writer가 확정한 카피를 통합하여, 사용자가 시스템의 실패 상황에서도 플랫폼에 대한 **신뢰감**을 잃지 않도록 시각적/경험적으로 일관성을 유지하는 최종 디자인 사양서입니다.\n\n--- \n\n## 🎨 1. 핵심 비주얼 가이드라인 (재확인)\n*   **Deep Blue (#004D66):** 플랫폼의 핵심 가치, 전문성, 신뢰(Primary Color). 메인 헤더, CTA 버튼 등 중요 정보를 강조.\n*   **Growth Green (#3CB371):** 긍정적 결과, 성공적인 행동 유도 (Success State). 최종 결제 완료, 목표 달성 시 사용.\n*   **Warning Orange/Red:** 경고 및 실패(Failure State). 사용자 입력 오류, 시스템 오류 발생 시 사용. **단, '시스템 실패'는 감정적 공감을 통해 해결해야 함.**\n\n## 💻 2. 컴포넌트별 상태 정의 (State Definition)\n\n| 컴포넌트 | 기본 상태 (Success) | Warning State | Failure State (Fallback) |\n| :--- | :--- | :--- | :--- |\n| **결제 버튼** | `[성공] 결제하기` (Growth Green)<br>→ 즉시 다음 단계로 이동\n| `재시도 필요` (Warning Orange) / 횟수 표시\n| `결제 실패. 잠시 후 다시 시도해주세요.` (Deep Blue 배경, Soft Red 텍스트)\n| **에러 메시지** | *(없음)* | *데이터 입력 오류:* "필수 항목을 확인해주세요." (경고 아이콘)\n| **시스템 알림** | *(성공적으로 처리됨)* | *임시 접속 불가:* "현재 시스템 점검 중입니다. [점검 예상 시간]에 다시 시도해 주세요."\n| **금융 연동 실패** | *(최적의 경로로 연결)* | *제휴사 데이터 전송 오류:* "데이터 전송에 문제가 발생했습니다. 잠시 후 다시 시도해주세요."\n| **결과 페이지 (MetricCard)** | `매출 증가율: +12%` (Growth Green) / 명확한 수치 제시\n| `데이터 조회 실패:` (Warning Orange) / "조회된 데이터가 없습니다." 텍스트 표시.\n| **Fallback UX** | *(사용자에게 안정감 제공)* | **[핵심] 금융 예외 상황 발생 시:** 단순 에러 코드 노출 금지. **'원인 설명 + 해결책 제안 + 신뢰 문구'**의 3단 구조 사용. (예: "잠시 시스템 문제로 결제가 안 됐지만, [대체 방안 A]를 통해 도와드리겠습니다.")\n\n## 📄 3. 최종 와이어프레임 상세 구현 스펙 (Mock-up Spec)\n\n### A. 성공 경로 시나리오 (Goal Achieved - Green Focus)\n*   **와이어프레임 섹션:** ROI 시뮬레이션 결과 요약 / 성장 로드맵 제시.\n*   **시각화 요소:** 최종 목표 달성치(MetricCard)는 **Growth Green**으로 강조. '성장'이라는 감정적 보상을 극대화합니다.\n\n### B. 실패 경로 (Failure State - Trust Focus)\n*   **와이어프레임 섹션:** 결제 직전/결과 페이지 전체 오버레이.\n*   **디자인 원칙:** **Deep Blue**의 배경을 유지하여 플랫폼 자체의 신뢰성을 잃지 않게 합니다. 실패 메시지는 너무 자극적이지 않은, 공감 기반의 카피를 사용합니다.\n*   **최종 Mock-up 구조 (예시):**\n    1.  **(헤더):** "결제에 문제가 발생했습니다." (딥 블루 배경)\n    2.  **(본문/공감):** *[Writer 확정 문구]* "잠시 불편을 드려 죄송합니다. 시스템 오류일 수 있으나, 저희 BDS는 고객님의 성공을 위해 [대체 방안]으로 도와드릴 준비가 되어 있습니다."\n    3.  **(액션 버튼):** `1. 재시도하기` (Deep Blue 배경의 CTA) / `2. 1:1 상담 요청` (보조 링크)\n\n## ✅ 개발팀 전달 사항 및 검토 포인트\n*   모든 예외 상황(API, 네트워크, 서버)은 **최소한 'Fallback UX' 구조**를 따르도록 로직을 설계해야 합니다.\n*   에러 코드를 절대 사용자에게 노출하지 말고, 오직 정의된 카피와 시각적 가이드만 사용합니다. \n\n--- \n\n***이 사양서는 BDS 플랫폼의 최종 UI/UX 기준점이 됩니다.***\n
