# 🎨 BDS 플랫폼: Trust Widget & PainGauge 시각적 일관성 검증 체크리스트 (V1.0)

**작성일:** 2026-06-13  
**작성자:** Designer (Lead Designer)  
**참조 문서:** `BDS_Final_Design_Spec_v1.0.md`, TrustWidget/PainGauge 상태 매트릭스 V2.0, 디자인 시스템 토큰 V2.0

---

## 🎯 목표 및 범위

본 체크리스트는 데이터 기반의 컴포넌트(Trust Widget, PainGauge)가 다양한 상태를 거치며 **시각적으로 일관성**을 유지하는지 검증하기 위함입니다. 단순히 기능 구현 여부를 넘어, 모든 인터랙션과 경계 조건에서 디자인 시스템이 정의한 시각적 규칙을 따르는지 확인해야 합니다.

## 🛠️ 검증 영역별 체크리스트

### 1. 컬러 및 상태(State) 매핑 일관성 (Color & State Consistency)

| 항목 | 테스트 케이스 / 상태 변화 | 기대되는 비주얼 스펙 (Design Spec Reference) | 개발팀 검증 포인트 (Implementation Check) |
| :--- | :--- | :--- | :--- |
| **🟢 최적/안전 구간 (A)** | Trust Score 80점 이상, Pain Level Low | 메인 컬러 (`--color-primary`) 기반의 *밝고 안정적인* 배경 및 아이콘. 경고색 요소 배제. | A 상태 진입 시 색상 코드가 정확히 토큰을 따르는가? 하이라이트 애니메이션이 과하지 않은가? |
| **🟡 주의/개선 필요 구간 (B)** | Trust Score 50~80점, Pain Level Medium | 경고색 (`--color-warning`)의 *채도와 투명도를 조절*하여 사용. 핵심 수치(Score)만 강조하고 배경은 중립적 유지. | B 상태로 전환 시 색상 변화가 점진적인지? (Hard Jump 방지). 텍스트 대비(Contrast Ratio)는 적정 수준을 유지하는가? |
| **🔴 위험/개입 필요 구간 (C)** | Trust Score 50점 미만, Pain Level High | 경고색 (`--color-danger`)의 *강한 존재감* 확보. 전용 에러 메시지 영역 활성화. 아이콘과 수치에 가장 강하게 적용. | C 상태 진입 시 사용자에게 명확하고 즉각적인 '행동 유도(CTA)'가 가능한 디자인이 유지되는가? (예: 버튼 강조) |
| **🚨 로딩/오류 상태** | API 통신 실패, 데이터 없음 (`Fallback`) | 시스템 기본 토큰 색상 사용. 메시지는 직관적이고, 컴포넌트의 *뼈대 레이아웃*은 그대로 유지되어야 함. | Skeleton Loader가 진입하는 페이지의 전반적인 여백(Padding/Margin)을 해치지 않는가? 에러 코드가 일반 사용자에게 노출되지 않도록 처리되었는가? |

### 2. 타이포그래피 및 레이아웃 무결성 (Typography & Layout Integrity)

| 항목 | 검증 영역 | 필수 확인 사항 | 체크 리스트 (Pass/Fail) |
| :--- | :--- | :--- | :--- |
| **헤드라인 H1** | 컴포넌트 최상단 타이틀 (`PainGauge Title`) | `DesignSystem_H1` 폰트, 크기(`2rem`), 굵기(Semi-Bold)가 상태 변화와 무관하게 고정되어야 함. | [ ] |
| **데이터 수치** | Score/Level 값 표시 (예: "78점", "Medium") | `DesignSystem_DataValue` 스펙을 준수하며, 글자 크기(Scale-2) 및 폰트가 명확해야 함. 상태별로 *크기 변화는 없어야* 함. | [ ] |
| **레이아웃 구조** | 컴포넌트 전체 배치 (Trust vs Pain) | Trust Widget과 PainGauge 간의 수평/수직 여백(Grid Gap) 및 분할선(Divider) 디자인이 일관되어야 함. (반응형 기준 필수). | [ ] |
| **사용자 피드백** | Helper Text / 가이드 문구 | 상태에 따라 표시되는 설명 텍스트는 `DesignSystem_Body`를 따르며, 최대 3줄을 넘기지 않도록 제어되어야 함. | [ ] |

### 3. 인터랙션 및 애니메이션 (Interaction & Animation)

| 항목 | 시나리오 | 기대 동작 스펙 (Expected Behavior) | 개발팀 검증 포인트 (Dev Check) |
| :--- | :--- | :--- | :--- |
| **상태 전환** | A $\rightarrow$ B 또는 B $\rightarrow$ C로 변화할 때 | 색상이 갑자기 변하는 것이 아니라, `Ease-out` 함수를 적용한 부드러운 트랜지션(Transition)을 가져야 함. (시간: 0.3s). | CSS/JS 애니메이션 코드가 명시된 타이밍과 정확히 일치하는가? 성능 저하 없이 매끄럽게 작동하는가? |
| **호버 효과** | 마우스를 컴포넌트 위로 올릴 때 | 강조(Focus)되는 영역만 미세하게 밝아지거나, 밑줄 애니메이션을 적용. 전체 레이아웃이 흔들려서는 안 됨. | 호버 시 발생하는 오버플로우(Overflow)나 레이아웃 이동 현상은 없는가? (특히 모바일 환경). |
| **데이터 업데이트** | 주기적인 데이터 새로고침 (`Refresh`) | 수치 변화는 *카운트업/다운* 애니메이션을 적용하여 '진행 중'의 느낌을 주어야 하며, 단순히 숫자가 뚝 바뀌면 안 됨. | 카운팅 로직이 부드럽게 작동하며, 너무 빠르게 깜빡이지 않는가? (디자인 가이드라인 준수 확인). |
